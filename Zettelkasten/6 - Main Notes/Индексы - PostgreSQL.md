# Индексы - PostgreSQL

```sql
SELECT      -- Просмотр всех индексов БД
	tablename, 
	indexname, 
	indexdef
FROM pg_indexes 
ORDER BY tablename;
```


---

В индексах PostgreSQL хранятся пары **ключ + CTID** — системный идентификатор кортежа (TID), который указывает на физическое расположение строки в файле таблицы (страница + смещение). 

При новой записи данных, старый указатель не удаляется сразу - за его удаление отвечает VACUUM.

```
B-tree индекс:
├── ключ (им индексируемого столбца)
└── CTID (ItemPointerData): 
    ├─ block_number (номер 8KB страницы)
    └─ offset_number (позиция строки на странице)
```

После получения конкретного адреса - смотрит на текущие `txmin`, `txmax` - проверяет, видима ли запись.

Если запись не видна, то смотрит следующую подходящую.

> [!warning] 
> Старые записи не удаляются при вставке новых, если она больше не видна ни в одной транзакции - удаляется VACUUM процессом.

Индекс хранит ссылку всегда на самую старую корневую запись - при поиске значения - просматривает все версии с данным ключом как в LinkedList.


## Поиск по индексу

1. **B-tree поиск** находит нужный ключ → получает **CTID** (системный идентификатор кортежа (TID))
    
2. **Heap-доступ** по CTID читает конкретную версию строки с диска
    
3. **Visibility check**: проверяет, видима ли версия для текущей транзакции
    
    - Если `xmin` committed и не superseded → используем
        
    - Если "мертвая" → HOT-обновление  или vacuum

> [!note]
> HOT - heat-only-tuple - оптимизация получения данных из таблицы. Вместо обновления таблицы с 
> Условия:
> - если индекс построен по id, транзакция не должна менять id. 
> - на странице где лежит старая версия должно быть место для новой записи

|Тип индекса|**Когда использовать**|**Операторы**|**Размер**|**Скорость записи**|
|---|---|---|---|---|
|**`B-tree`** (по умолчанию)|**=**, **<**, **>**, **BETWEEN**, **IN**, **IS NULL**|`=`, `<>`, `<`, `>`, `<=`, `>=`, `BETWEEN`|Средний|Быстрая|
|**`Hash`**|**Только точное =** (с PG 10+)|`=`|Малый|Очень быстрая|
|**`GiST`**|Геоданные, полнотекст, диапазоны|`&&`, `@>`, `<@`, `~=` (PostGIS, tsvector)|Большой|Медленная|
|**`SP-GiST`**|Кластеризованные/географические данные|IP-адреса, геохэши, квадродеревья|Малый|Быстрая|
|**`GIN`**|**Массивы**, **JSONB**, **полнотекст** (`@@`)|`@>`, `&&`, `ts @@`|**Очень большой**|**Очень медленная**|
|**`BRIN`**|**Огромные отсортированные таблицы**|Диапазоны на блоках|**Минимальный**|Максимальная|
|**`Bloom`** (расширение)|Композитные запросы с низкой селективностью|`&&` (множественные условия)|Малый|Быстрая|

```sql
-- Стандартный B-tree (по умолчанию)
CREATE INDEX idx_users_email ON users(email);

-- Hash (только =)
CREATE INDEX idx_users_id_hash ON users(id) USING HASH;

-- GIN для JSONB поиска
CREATE INDEX idx_users_data ON users USING GIN (data);

-- GIN для полнотекста
CREATE INDEX idx_docs_fts ON docs USING GIN (to_tsvector('russian', content));

-- GiST для геоданных (PostGIS)
CREATE INDEX idx_places_geom ON places USING GIST (geom);

-- BRIN для временных рядов (огромные таблицы)
CREATE INDEX idx_logs_timestamp ON logs USING BRIN (timestamp);

-- SP-GiST для точек
CREATE INDEX idx_points ON points USING SPGIST (location);
```

## Два разных сценария обновления

В PostgreSQL существуют два механизма обновления строк, и поведение индекса зависит от того, какой сработал:

### 1. Обычный UPDATE (не HOT)

Когда обновляются колонки, входящие в индекс, срабатывает обычный механизм:

1. Создается новая версия строки в heap (может быть на другой странице)
2. В индекс добавляется НОВАЯ запись с TID новой версии
3. Старая запись в индексе остается висеть (мертвой) до VACUUM

В результате в индексе одновременно существуют обе записи — старая (указывает на старую версию) и новая (указывает на новую версию). Индекс-скан найдет обе, но MVCC-visibility check отфильтрует невидимую для текущей транзакции.

### 2. HOT UPDATE

Когда обновляются только неиндексированные колонки, срабатывает HOT:

1. Новая версия создается в heap на той же странице
2. В индекс НИЧЕГО НЕ ДОБАВЛЯЕТСЯ
3. Индекс продолжает ссылаться на КОРНЕВУЮ версию (которая помечена HEAP_HOT_UPDATED)
4. PostgreSQL при чтении идет по цепочке t_ctid до актуальной версии

Здесь в индексе всегда одна запись на всю цепочку версий — она указывает на корневую (самую первую) версию в цепочке.

оводу «устаревшей» версии — я имел в виду именно HOT-сценарий. В этом случае:

- Индекс ссылается на корневую версию, которая с точки зрения бизнес-данных уже «неактуальна» (там старые значения неиндексированных полей)
- Но для индекса это не важно, потому что индексированные значения в новой версии те же самые
- PostgreSQL просто следует по цепочке и находит актуальную версию с нужными значениями неиндексированных полей

> [!note] 
> Когда он ищет актуальную запись - проходит в heap и смотрит, жива ли запись. Если запись уже мертва - пометит её в индексе `LP_DEAD` бит, что запись мертва.


В случае обычного UPDATE (когда меняется индексированная колонка) — в индексе появляется новая запись, указывающая на новую версию, так что индекс ссылается уже на «свежую» версию.

Таким образом, фраза «в индексе всегда хранится ссылка на устаревшую корневую версию» верна только для HOT-цепочек, но не для обычных обновлений индексированных полей. Извини за путаницу.



## Особенности

- **Индекс НЕ хранит данные** — только ключи + CTID
    
- **MVCC полностью на heap-уровне** — индекс не знает о версиях
    
- **Dead Tuples**: индекс указывает на "мертвые" версии, пока не сработает VACUUM
    
- **Bloom фильтры/кэш**: ускоряют visibility check для горячих страниц

	## **Как работает поиск при обычном UPDATE**
	
	Когда в индексе есть обе записи (старая и новая версия) и ты выполняешь запрос с условием, которому соответствуют обе:
	
	1. Индекс-скан находит обе записи — он видит две TID в индексе, обе проходят по условию поиска по индексированной колонке.
	
	2. Доступ к heap: PostgreSQL идет в heap по каждой TID, читает заголовок строки с xmin/xmax и выполняет MVCC-visibility check — определяет, видна ли эта версия текущей транзакции.
	
	3. Фильтрация: Возвращается только та версия, которая видна. Если текущая транзакция началась до UPDATE — видна старая версия, если после — новая.
	
	## **Про установку флага LP_DEAD**
	
	Флаг LP_DEAD устанавливается не для всех невидимых версий, а только для тех, которые оказались "мертвыми для всех" (dead to all). Это происходит когда:
	
	- PostgreSQL читает строку из heap и видит, что она удалена (есть xmax)
	- Проверяет, что транзакция с xmax уже завершена
	- Проверяет, что xmax меньше xmin всех активных транзакций (т.е. эта версия точно не нужна никому)
	
	Только в этом случае на индексную запись, указывающую на эту строку, ставится флаг LP_DEAD. Если же старая версия еще видна какой-то длительной транзакции (например, бэкап или аналитический запрос), флаг не ставится — она пока "жива" для них.
	
	Таким образом, индекс-скан возвращает актуальную для текущей транзакции версию, а флаг мертвым ставит только той записи в индексе, которая указывает на версию, ставшую ненужной вообще всем транзакциям.
