# Volatile - Multithreading

С точки зрения Java все переменные (за исключением локальных переменных, объявленных внутри метода) хранятся в главной памяти, которая доступна всем потокам, кроме этого, каждый поток имеет локальную - рабочую память, где он хранит копии переменных, с которыми он работает, и при выполнении программы поток работает только с этими копиями.

Надо отметить, что это описание не требование к реализации, а всего лишь модель, которая объясняет поведение программы, так, в качестве локальной памяти не обязательно выступает кэш память, это могут быть регистры процессора или потоки могут вообще не иметь локальной памяти.





---
### `volatile`

Запись `volatile` переменных производится в основную память, минуя локальную. и чтение `volatile` переменной производится также из основной памяти, то есть значение переменной не может сохраняться в регистрах или локальной памяти потока и операция чтения этой переменной гарантированно вернёт последнее записанное в неё значение.

Запись volatile:
  - «Сбрасывает» все предыдущие записи этого потока в память.
  - Затем записывает новое значение volatile.

Чтение volatile:
  - «Подтягивает» из памяти все записи, сделанные потоками до их записи в volatile.
  - Затем читает актуальное значение volatile.

Поэтому volatile хорошо подходит для:
  - Флагов остановки (stopRunning).
  - Публикации неизменяемых объектов после инициализации.

Но volatile НЕ даёт атомарности для составных операций (x++, check-then-act, read-modify-write). Для этого нужны:
  - synchronized/Lock, или
  - атомарные классы (AtomicInteger и др.).

> [!note]
> Вместе с volatile переменной поток синхронизирует и все остальные переменные с heap:
> - при чтении volatile - обновит значения не volatile локальных переменных потока из Heap
> - при записи volatile - flushes значения всех переменных в heap (даже тех, что не volatile)

#### Happens before guarantee для volatile

Запись в volatile‑переменную **happens‑before** each subsequent read этой же переменной в другом потоке.

Это означает только:

- Если запись уже произошла и затем случилось чтение, читающий поток увидит значение этой записи и все обычные записи до неё.
    
- Но порядок «кто раньше — read или write» решается реально временем выполнения, JMM не заставляет запись произойти раньше чтения

**~={red}Volatile не заставляет ждать пока поток A не завершит работу с volatile переменной и рзрешает потку В считать старое значение.=~ -> возможен [[ACID principles - БД#Аномалии чтения данных|lost update]]**

#### Супер подробно, если тупишь:

- Если поток A читает volatile, а поток B в это время тоже читает volatile — оба просто читают «последнюю записанную» версию; никакого ожидания или блокировки нет.
    
- Если поток A собирается что‑то сделать и потом записать volatile, а поток B уже читает volatile сейчас, JMM не говорит: «подожди, пока A запишет». B просто увидит текущее опубликованное значение (предыдущую запись, если новой ещё не было).
    
- Если поток A вообще никогда не пишет (только читает и использует в вычислениях), он просто читает актуальное на момент чтения значение; никаких дополнительных happens‑before цепочек от него не возникает, кроме его собственных чтений.

### `final`

Есть одна проблема, связанная с `final` полями: реализация разрешает менять значения таких полей после создания объекта (это может быть сделано, например, с использованием механизма reflection). Если значение `final` поля—константа, чьё значение известно на момент компиляции, изменения такого поля могут не иметь эффекта, так-как обращения к этой переменной могли быть заменены компилятором на константу. 
Также спецификация разрешает другие оптимизации, связанные с `final` полями, например reordering: операции чтения `final` переменной могут быть переупорядочены с операциями, которые потенциально могут изменить такую переменную. Так что рекомендуется изменять `final` поля объекта только внутри конструктора, в противном случае поведение не специфицировано.
