# PostgreSQL - JSON types


- JSON - точная текстовая копия
- JSONB - двоичный формат исходного JSON
	`+` компактный;
	`+`  поддерживает индексы на вложенные поля;
	`+` Чтени и поиск по объекту внутри документ значительно быстрее благодаря предварительной разборке и поддержке индексов;
	`-` накладные расходы при вставке.

При вставке JSON нет накладных расходов на сериализацию. При вставке JSONB исходный документ анализируется, строится граф документа и преобразуется в двоичный формат.

> [!note] 
> Если нам нужно сравнивать документ хранимый в БД с каким-то получаемым документом - JSONB не подойдёт, т.к. он чаще всего меняет порядок полей в документе, может убирать дубликаты.


---
### **Работа с JSON/JSONB**

- Поддержка индексов (GIN, GiST) на поля JSONB, ускоряющих поиск и агрегацию.

- Удобство для манипуляций: обновления частей объектов без перезаписи всей структуры, эффективные операции над данными.

- Упрощённые операции сравнения и объединения данных внутри JSONB.

Пример операций с JSONB:
	
```sql
-- Исходные данные
INSERT INTO test (id, data) VALUES 
(1, '{"name": "Alice", "age": 30, "tags": ["a", "b"]}');
	
-- Обновить возраст (точечное изменение)
UPDATE test 
SET data = jsonb_set(data, '{age}', '35') 
WHERE id = 1;
	
-- Добавить/обновить вложенный объект, создавая путь если отсутствует
UPDATE test 
SET data = jsonb_set(data, '{address, city}', '"Moscow"', true) 
WHERE id = 1;
```

---
### Рекомендации
	
- Используйте JSONB по умолчанию для новых проектов, если планируете частый поиск, фильтрацию и индексы внутри JSON.
    
- Рассматривайте JSON только если вам критично сохранить точный текст входного JSON или данные редко требуют внутреннего анализа.

### Пример сценариев
	
- Приложение, которое хранит конфигурационные данные с частыми запросами по полям внутри JSON: JSONB.
    
- Логовая система, где каждый вход сохраняется как чистый JSON строкой и редкие внутренние запросы: можно рассмотреть JSON.
