# Проблема N+1

Проблема N+1 - проблема подтягивания зависимых сущностей каждой записи при FetchType.LAZY. 

> [!note]
> Установка FetchType.EAGER - не поможет. Так и останется N+1 запросов. Только теперь запросы будут делаться сразу после получения списка (списка Person в данном случае)


![[Pasted image 20251108154607.png]]

Т. е., чтобы получить всех Person и связанные Items вручную потребуется сделать N+1 запросов. (1 запрос - на получение всех Person, и по одному запросу на получение связанных Items для полученных ранее N людей).


---
### Решение проблемы:

```java
Set<Person> people = new HashSet<>(session.createQuery("FROM Person p LEFT JOIN FETCH p.items", Department.class).getResultList());
```

~={pink}Зачем используем Set?=~

- Если у Person 10 предметов, то в результирующем наборе будет **10 строк** с одинаковыми данными человека, но разными данными предметов.
    
- Hibernate при маппинге создаёт один объект `Person`, но возвращает его столько раз, сколько строк пришло из SQL.
    
- В итоге `List<Person>` содержит повторяющиеся ссылки на один и тот же объект Person1 столько раз, сколько у него items (причем все items будут у каждого из них, т.е. это полностью идентичные объекты).

```
Person1 | ItemA
Person1 | ItemB
Person1 | ItemC
```

**Hibernate‑уровень**

- Hibernate создаёт **один объект** `Department` для каждой уникальной строки по первичному ключу.
    
- При обработке следующих строк с тем же `Department.id` он не создаёт новый объект, а добавляет новых разработчиков в коллекцию `developers` этого департамента.
    
- Поэтому коллекция внутри департамента постепенно наполняется всеми связанными сущностями.

**Несмотря на то, что Hibernate правильно наполняет коллекцию, он всё равно возвращает департамент столько раз, сколько строк было в SQL.**
