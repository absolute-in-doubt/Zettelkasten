Q: У нас есть один сервис, который отправляет сообщения (события) в Kafka и 3 сервиса, которые их читают. Стоит ли отправлять сообщения в топик используя определённый ключ? Будут ли все Consumers работать в этом случае?
A: Если у нас несколько Consumers, то они читают каждый свою partition of the given Topic. (Только lead partitions). Елис отправлять сообщения с определённым ключом, то все события будут отправляться только в один определённый топик. -> сообщения будут приходить постоянно одному и тому же Consumer -> остальные 2 простаивают.
	
Если же мы будем отправлять сообщения без определённого ключа, Kafka сама будет по своим алгоритма распределять сообщения по partitions -> все Consumer будут задействованы.
	
> **NB!** Если отправляем без ключа - это не значит, что отправляем сразу во все partitions, это значит, что нужный partition, куда отправить это событие, выберет Kafka.
>
> Свой ключ нужен не константный. Если у нас имеется несколько сообщений, порядок которых имеет значение - отправляем их с одним и тем же ключом. Но меняем ключ между группами таких сообщений.
	
**Пример:**
	
Если работаем с пользователями - как ключ логично использовать `session_id`. -> Все действия пользователя упорадочены в одной partition.
	
Если цель - аналитика однотипных действий - можно по action_type. -> Все однотипные действия хранятся по порядку в одном partition.

Q: Опиши всю логику отправки и получения сообщений через Kafka (Producers > 1. Consumer > 1).
A: Количество Producers не особо важно. Каждый Producer оправляет сообщение в топик (если мы не реализуем свой алгоритм распределения между partitions - тогда юзаем ключи). Если отправляем просто в Topic - Kafka сама выбирает **один** partition, куда запишет отправленное соообщение.
	
Каждый Consumer читает из своего partition. В описанной модели отправленное сообщение обработает тот Consumer, в partition которого придёт сообщение.
	
> **NB!** Kafka каждый раз будет отправлять сообщения в разные partitions, таким образом, каждый Consumer видит только свои сообщения.
	
> **NB!** При чем, если у нас несколько Consumer Groups, то Consumer из одной группы shares the partition с Consumer из другой группы.

Q: Как Consumers отслеживают прочитанные записи? Что такое offset и где он хранится?
A:  **Kafka хранит offsets для Consumer Group в специальном внутреннем топике `__consumer_offsets`**.
	
```
__consumer_offsets — компактный топик (50 partitions по умолчанию)
Ключ: group_id + topic + partition
Значение: offset + metadata + timestamp + leader_epoch
```
	
Пример (упрощённо):
	
```
Consumer Group: "order-processors"
Topic: orders (3 partitions)
	
__consumer_offsets содержит:
[group=order-processors, topic=orders, partition=0] → offset=150
[group=order-processors, topic=orders, partition=1] → offset=89  
[group=order-processors, topic=orders, partition=2] → offset=234
```
	
Offset - номер последней записи, короую обработал данный Consumer в данном Topic .
