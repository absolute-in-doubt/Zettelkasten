# Виды GC в HotSpot JVM - JMM


Основные сборщики мусора HotSpot:​

#### **Serial GC**

- Простой однопоточный, stop‑the‑world, подходит для маленьких heap и утилит.

- Включается флагом `-XX:+UseSerialGC`.​

- **Как работает:**
    
    - Когда young generation заполняется - живые объекты собираются копированием (Eden → Survivor → Old). ~={cyan}Если в процессе копирования живых объектов To space переполняется, то оставшиеся живые объекты из Eden и From space, которым не хватило места в To space, будут перемещены в Old generation, независимо от того, сколько сборок мусора они пережили.=~
        
    - Old generation — mark-sweep‑compact: помечаем живые, сдвигаем их к началу кучи, освобождаем хвост.​
		
	- **Full GC** — проводит маркировку и очистку **всего heap-а**: young + old.  Запускается при заполнении Old generation.
        
    - Все фазы выполняются одним потоком, пока GC работает — приложение стоит.

| Тип GC       | Когда запускается                                                        | Область                        | Stop-the-world                |
| ------------ | ------------------------------------------------------------------------ | ------------------------------ | ----------------------------- |
| **Minor GC** | Eden заполнен (нет места для нового объекта)                             | Только Young (Eden + Survivor) | Да, весь хип не затрагивается |
| **Full GC**  | Tenured (Old) заполнен **ИЛИ** Survivor не может принять объекты из Eden | Вся куча (Young + Old)         | Да, полная остановка.         |


---
#### **Parallel GC (Throughput Collector)**

- Многопоточный, ориентирован на максимум пропускной способности, допускает долгие паузы.

- `-XX:+UseParallelGC`.​

- **Как работает:**
    
    - Young GC (Minor) — параллельная копирующая сборка young‑generation (несколько потоков обрабатывают Eden+Survivor).​
        
    - Old GC (Full) — параллельный mark‑sweep-compact: помечаем живые в old, затем компактируем.​
        
    - Всё равно stop‑the‑world, просто пауза короче за счёт многопоточности.


#### **G1 (Garbage‑First) GC**


**G1 GC** (Garbage First) — generational low-pause collector (с Java 7, по умолчанию с Java 9+). Разбивает heap на **регионы** (1–32 МБ), собирает **young** (Eden + Survivors) и **old** по мере "грязности" (по liveness, RSet), стремясь к паузам ≤ `MaxGCPauseMillis` (200 мс по умолчанию)

- `-XX:+UseG1GC`.​

Фазы G1 GC: 

1. Minor GC + Initial Markup - копирующий GC, выполняется при заполенении памяти до определенного порога очистка young generation регионов. Запускается при заполнении Eden (~50% young). Определяет **Initial Mark** (корни: потоки, JNI, статические поля).
   ~={orange}Young GC **включает** Initial Mark (STW), но это не "Minor GC +", а единая пауза=~


2. **Root Region Scan**: Конкурентно сканирует Survivor-регионы на ссылки в old (для Concurrent Mark). Завершается перед следующим Young GC.
    
3. **Concurrent Mark**: Конкурентно (параллельно приложению) строит **liveness** регионов (SATB-алгоритм). Прерывается Young GC. Завершается **Remark** (STW: сливает SATB-батчи, финальная пометка).
    
4. **Cleanup**: STW/конкурентно: подсчёт свободного места, возврат пустых регионов в free list.
    
5. **Mixed GC**: После Mark (если old заполнен > `-XX:InitiatingHeapOccupancyPercent=45%`): STW evacuation **old + young** регионов (по "грязности"). До Full GC.
    

**Full GC**: STW, если mixed не справляется (редко).
1. Remark + Mixed GC - после завершения Concurrent Markup - происходит STW пауза, завершается пометка живых объектов (Remark) и запускается копирующий коллектор, который копирует из наиболее грязных регионов всех типов в более чисты регионы следующего типа, например из S0 в S1, из S1 в Old. Из самых грязные Old generation regions данные копируются в более чистые регионы и очищенные регионы превращаются в Eden regions

**Параметры**: `-XX:MaxGCPauseMillis`, `-XX:G1HeapRegionSize`, `-XX:InitiatingHeapOccupancyPercent`.

| Тип GC       | Триггер запуска                                                                                 | Области сборки                                       |
| ------------ | ----------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| **Minor GC** | Запускается при заполнении Eden (~50% young).                                                   | Только Young регионы (Eden + Survivor).              |
| **Mixed GC** | Concurrent marking завершён **И** Old Gen > `InitiatingHeapOccupancyPercent` (45% по умолчанию) | Young + N Old регионов с большим количеством мусора. |

#### **ZGC (Z Garbage Collector)**

- Современный, практически паузонезависимый, работает с очень большими кучами, паузы ~до 10 мс.

- `-XX:+UseZGC`.​

- **Ключевые механизмы:**
	
	- Многоверсионные указатели (colored / tagged pointers) и read‑/write‑barriers позволяют двигать объекты **конкурентно**, пока приложение работает.​
		
	- Фазы:
		
		- **Concurrent Mark**: как у G1 — помечаем живые объекты, в основном конкурентно.
			
		- **Concurrent Relocate (Evacuation)**: объекты копируются между регионами, ссылки обновляются конкурирующими потоками с помощью барьеров.​
			
		- STW‑паузы очень короткие и используются лишь для маленьких служебных шагов (например, root scanning init, remap start).
		
- **Особенности:**
	
	- Куча тоже разделена на регионы, но ZGC изначально разрабатывался как **негеренерационный**, позже появился generational ZGC (Java 21+), добавляющий поколенческую оптимизацию.


#### **Shenandoah GC**

- Конкурентный низкопаузный GC от Red Hat, региональный, похожей идеи с ZGC, но с другими реализациями эвакуации и обновления ссылок

- `-XX:+UseShenandoahGC`.​

- **Как работает (очень кратко):**​
	
	- Тоже региональный и поколенческий (в новых версиях).
	    
	- Основные фазы:
	    
	    - **Concurrent Marking**: конкурентное помечание живых объектов.
	        
	    - **Concurrent Evacuation**: копирование живых объектов из выбранных регионов в другие (эвакуация) конкурентно с приложением.
	        
	    - **Concurrent Update Refs**: конкурентное обновление всех ссылок на перемещённые объекты (отличие от ZGC: ссылки обновляются не сразу при доступе, а отдельной фазой).​
	        
	    - Короткие STW‑фазы используются для синхронизации (init mark, final mark, init update refs и т.п.).
	    
	- Активно использует барьеры чтения/записи и механизмы pacing/degeneration, чтобы или удерживать паузы короткими, или в крайних случаях перейти во временный STW‑режим.



#### **Epsilon GC**

-  Это экспериментальный сборщик мусора, введённый в Java 11, который не освобождает память, а только выделяет её, пока не возникнет OutOfMemoryError. Он обеспечивает минимальный overhead на GC, полностью исключая паузы от сборки мусора.​

**Основные применения**

- Тестирование производительности: позволяет измерить скорость приложения без влияния GC.​
    
- Оценка memory pressure: показывает объём выделяемой памяти без очистки.
    
- Короткоживущие задачи: идеален для программ, завершающихся до исчерпания heap (например, скрипты).

- `-XX:+UseEpsilonGC`.

## Что ещё важно помнить

- Все перечисленные сборщики (кроме Epsilon, понятн дело) — **поколенческие** (или имеют поколенческий режим), то есть разделяют объекты по возрасту и по‑разному собирают young и old поколения.​
    
- Отличия в основном по трём осям:
    
    - **STW vs concurrent** (сколько работы делается с остановкой мира).
        
    - **Serial vs parallel** (сколько потоков использует GC).
        
    - **Region‑based vs contiguous** (работа с регионами и локализация фрагментации)
