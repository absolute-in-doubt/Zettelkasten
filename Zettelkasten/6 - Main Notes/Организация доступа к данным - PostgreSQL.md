# Организация доступа к данным - PostgreSQL


### Sequential scan

![[Pasted image 20260214174330.png]]

Подойдет, когда нужно считать почти всю таблицы.

`+` Если табличка маленька 10-100 записей - она уместиться в 1-2 страницы -> sequential scan будет самым эффективным. Какие бы индексы не добавляли - всё равно на `EXPLAIN`,  ``EXPLAIN ANALYZE`` получим sequential scan.

`-` Хуйня, если надо прочитать небольшую подвыборку

### Index Scan


![[Pasted image 20260214175314.png]]

Сканируем по индексу (B-tree)

`-` часто бегает в таблицу и что-то считывает -> не варик, если надо много считать.

Тут в третью страницу мы сходили три раза.

### Bitmap Heap Scan


Когда нужно выбрать большое количество строк. 
**Выбрано 5-20% строк таблицы** — это "золотая середина" между index scan и seq scan

`Bitmap Heap Scan` — это способ PostgreSQL **выбрать много строк через индекс, но прочитать только нужные страницы кучи последовательно**, вместо `Seq Scan` или частого рандомного чтения по `Index Scan`.


![[Pasted image 20260214175359.png]]

`+` идём в Heap единожды для каждого элемента

1. `Bitmap Index Scan` по индексу → в памяти битовая карта страниц/строк. Заполняем её за один проход.;
    
2. `Bitmap Heap Scan` → читает только помеченные страницы кучи и до‑фильтрует строки.


Первый проход - заполняем битовую маску. (со всем данными, и устаревшими и нет, игнорируя те, что отмечены как `LP_DEAD`)

Когда составили карту - просматриваем данные из heap.

### Index Only Scan

Суть в том, что вообще не ходит в Heap. Все значение столбцов должны быть частью индекса.

![[Pasted image 20260214180138.png]]

Частичный индекс - индекс по подвыборке.

Все данные, что мы ищем, просматриваем через [[Карта свободного пространства & карта видимости  - PostgreSQL|карту видимости]]. Если карта видимости подтверждает, что данные живые - не ходим в кучу искать актуальную версию. Если данные dead - ищем в heap актуальную версию.

Если найденный указатель указывает на страницу, которая в карте видимости указана как зелёная - возвращаем значение из кэша (буфера), если же красная - приходится идти в heap и проверять (тут только один раз пришлось)


> [!tip] Note
> Чтобы получение данных работало через Index Only Scan можно использовать:
> -  частичные индексы (например индекс по задачам, у которых статус "ToDo", т.к. они скорее всего будут alive):
> ```sql
> CREATE INDEX idx_name ON table(column) WHERE condition;  -- напр. WHERE status = 'active'
> ```
> - функциональные индексы:
> ```sql
> CREATE INDEX idx_lower_email ON users(LOWER(email));
> ```
> Планировщик выбирает Index Only Scan, если индекс содержит все запрашиваемые столбцы (включая `ctid` для MVCC) и `WHERE` совпадает с предикатом — без доступа к heap таблицы. Частичные/функциональные индексы подходят, т.к. плотно покрывают данные запроса.
