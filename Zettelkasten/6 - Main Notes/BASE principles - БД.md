# BASE (альтернатива ACID в NoSQL)

**BASE** — набор принципов, которыми часто руководствуются NoSQL-системы для достижения высокой доступности и горизонтальной масштабируемости.

Расшифровывается как:

- **B**asically **A**vailable
    
- **S**oft state
    
- **E**ventual consistency
    

---

## 1. Basically Available (базовая доступность)

Система стремится **всегда отвечать на запросы**, даже если часть узлов недоступна.

### Что это означает?

- При отказе некоторых серверов система продолжает работать.
    
- Пользователь почти всегда получает ответ.
    
- Но ответ может быть не самым свежим.
    

### Пример

Пользователь изменил имя профиля.

Сразу после изменения:

- сервер A уже содержит новое имя;
    
- сервер B ещё содержит старое.
    

Если запрос попадёт на B, пользователь увидит старые данные, но система всё равно ответит.

### Главное

> Предпочитается вернуть возможно устаревшие данные, чем не ответить вообще.

---

## 2. Soft State (мягкое состояние)

Состояние системы может изменяться **даже без новых запросов пользователя**.

Причина — фоновая синхронизация реплик.

### Пример

Есть три реплики:

```text
Replica1 -> "John"
Replica2 -> "John"
Replica3 -> "Jon"
```

Через некоторое время реплики автоматически синхронизируются:

```text
Replica1 -> "John"
Replica2 -> "John"
Replica3 -> "John"
```

Хотя пользователь ничего не делал.

### Главное

> Состояние системы не является строго фиксированным и может меняться со временем из-за репликации.

---

## 3. Eventual Consistency (согласованность в конечном итоге)

Если перестать изменять данные, то **через некоторое время все реплики придут к одному состоянию**.

### Пример

Пользователь изменил баланс:

```text
Node1 -> 100$
Node2 -> 80$
Node3 -> 80$
```

Через некоторое время:

```text
Node1 -> 100$
Node2 -> 100$
Node3 -> 100$
```

Все узлы станут согласованными.

### Важно понимать

В течение некоторого времени разные клиенты могут видеть разные данные.

Например:

```text
Клиент A -> баланс 100$
Клиент B -> баланс 80$
```

Это нормальное поведение для BASE-систем.

---

## Почему BASE появился?

Из-за ограничений теоремы CAP.

В распределённых системах при сетевых разделениях часто выбирают:

- **Availability** (доступность);
    
- **Partition Tolerance** (устойчивость к разделению сети);
    

жертвуя строгой согласованностью.

---

## BASE vs ACID

|ACID|BASE|
|---|---|
|Строгая консистентность|Консистентность со временем|
|Транзакции|Обычно ограниченные транзакции|
|Приоритет корректности данных|Приоритет доступности|
|Возможен отказ в обслуживании|Система старается отвечать всегда|
|Типично для SQL|Типично для NoSQL|

---

## Что важно запомнить для собеседования

> BASE — это подход, используемый многими NoSQL-системами, при котором ради высокой доступности и масштабируемости допускается временная несогласованность данных. Система гарантирует, что в конечном итоге все реплики синхронизируются (**eventual consistency**).

### Ключевая идея

> **BASE = Availability + Scalability за счёт ослабления строгой консистентности.**
