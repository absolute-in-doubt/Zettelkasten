
Theory for the cards: [[Сравнение в Java]]

Q: Какими свойствами должен обладать переопределенный метод equals?
A:   
**Обязательные свойства (контракт equals)**
	
- **Рефлексивность**: Объект всегда равен сам себе — `x.equals(x) == true`.
    
- **Симметричность**: Если `x.equals(y)`, то `y.equals(x)` — тоже `true`.​
    
- **Транзитивность**: Если `x.equals(y)` и `y.equals(z)`, то `x.equals(z)` — `true`.
    
- **Консистентность**: Многократные вызовы `equals` дают одинаковый результат, если объект не изменился.​
    
- **Сравнение с null**: `x.equals(null)` всегда `false`.
    
---
**Дополнительные рекомендации**
	
- **Совместимость с hashCode()**: Если `x.equals(y) == true`, то `x.hashCode() == y.hashCode()` (не наоборот обязательно).​
    
- **Проверка типов**: Сначала `if (this == obj) return true; if (obj == null || getClass() != obj.getClass()) return false;` — для производительности и безопасности.​
    
- **Сравнение полей**: Рекурсивно вызывать `equals` на полях (для объектов) или `==` (для примитивов).​
    
	
**Вывод**: Несоблюдение этих свойств приводит к багам в HashMap (дубликаты, потерянные ключи). Всегда переопределяйте `equals` вместе с `hashCode`.

Q: В каких случаях в equals() стоит использовать instanceof, а в каких сравнение через getClass()? Что предпочтительнее?
A:   
**instanceof**
	
- **Когда использовать**: Если равенство возможно между объектами суперкласса и **подклассов** (Liskov substitution). Подкласс `instanceof Superclass` вернёт true, позволяя сравнение.​
    
- **Пример случая**: Базовый класс `Shape` равен `Circle` (подклассу), если поля совпадают. Полезно для коллекций — ключи подтипов найдутся.​
    
- **Минусы**: Нарушает симметрию, если подкласс не переопределяет `equals` (A.equals(B) true, но B.equals(A) false). Риск "смешанных" равенств.
    
---
**getClass()**
	
- **Когда использовать**: Если объекты равны **только при идентичных классах** (строгий тип). `obj.getClass() == this.getClass()` исключает подклассы.​
    
- **Пример случая**: `Person` не равен `Employee` (подклассу), даже если поля совпадают — разные типы логики. Обеспечивает симметрию и транзитивность.​
    
- **Минусы**: Подклассы не равны суперклассу, что может нарушить LSP в коллекциях (ключ подкласса не найдёт суперкласс).​
    
---
***Что предпочтительнее?***
	
**getClass() предпочтительнее** в большинстве случаев: гарантирует контракт `equals` (симметрия, транзитивность), избегает сюрпризов с подклассами. Рекомендуют Joshua Bloch (Effective Java) и SonarQube для нефинальных классов. Используйте `instanceof`, только если дизайн явно позволяет равенство по иерархии и класс финальный/метод final.
