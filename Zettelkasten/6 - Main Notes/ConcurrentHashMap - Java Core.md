## Ключевые отличия от HashMap

| Аспект                  | HashMap                                 | ConcurrentHashMap                                            |
| ----------------------- | --------------------------------------- | ------------------------------------------------------------ |
| **Потокобезопасность**  | Нет, требует внешней синхронизации      | Да, поддерживает параллельный доступ                         |
| **Null ключи/значения** | Разрешены                               | Запрещены (null вызовет NPE)                                 |
| **Итерация**            | Простая, но небезопасная в multi-thread | Weakly consistent (итератор видит неснимутативные изменения) |
| **Размер по умолчанию** | capacity=16, loadFactor=0.75            | capacity=16, concurrencyLevel=16 (до Java 7)                 |

```
table: Node[] 

bucket[0] -> null
bucket[1] -> Node(key=A) -> Node(key=I) -> Node(key=Q)
bucket[2] -> TreeBin
             └─ красно-черное дерево из Node
bucket[3] -> ForwardingNode   // во время resize
bucket[4] -> Node(key=X)
```

##### Итератор

~={red}В `ConcurrentHashMap` итератор fail-safe (но `Weakly consistent`)**но работает не с копией как `CopyOnWriteArrayList`**. Итератор проходит **живую структуру** (массив `Node[]` с volatile полями). Он захватывает ссылку на таблицу в момент создания, но видит **некоторые** добавления/изменения после этого через volatile-семантику.=~

**Итератор ConcurrentHashMap НЕ гарантирует видимость всех изменений.** Weakly consistent означает непредсказуемое поведение: **может видеть некоторые изменения после создания, но не обязаны видеть все**.
	
**Что итератор гарантированно видит:**
	
- Элементы, существовавшие **на момент создания** итератора
    
- **Никогда не бросает** `ConcurrentModificationException`
    
**Что итератор МОЖЕТ видеть (не гарантировано)**
	
1. Добавленные элементы — если они появились ДО посещения их бакета 
2. Изменённые значения — через volatile-семантику
	
**Что итератор МОЖЕТ НЕ видеть:**
	
1. ✅ Удалённые элементы — если удалены ДО посещения их бакета
	
2. ✅ Новые элементы — если добавлены ПОСЛЕ посещения бакета
	
3. ✅ Изменения в других бакетах — зависит от timing

## Реализация синхронизации

**До Java 8**: Разделение на **сегменты** (по умолчанию 16). Каждый сегмент — это мини-HashMap с собственной блокировкой (ReentrantLock). Потоки блокируют только нужный сегмент, позволяя ~16 одновременных операций записи. Чтение — без блокировок (volatile).

**Java 8+**: Отказ от сегментов в пользу **CAS (Compare-And-Swap)** + **synchronized**.

- Структура: массив `Node[]` (как в HashMap) + volatile элементы.
    
- **put()**: 
	1. Вычисляется индекс bucket.
    
	2. Если bucket пустой, создаётся новый Node.
    
	3. Выполняется CAS: “если в ячейке всё ещё `null`, положи туда этот Node”.
    
	4. Если CAS успешен — вставка завершена без `synchronized`.
    
	5. Если CAS неудачен — значит, другой поток уже успел записать туда данные, и дальше используется более тяжёлый механизм синхронизации
    
- **get()**: ~={red}volatile-чтение **без синхронизации**.=~
    
- **Resize**: Помогают все потоки (cooperative resizing), перенося части таблицы. 
    
- Дополнительно: атомарные методы `putIfAbsent()`, `replace()` через CAS.​


**~={red}?!=~** **Когда CAS, а когда lock при `put()?

- **CAS** — для вставки в пустой bucket.

- **`synchronized`** — если bucket уже занят и нужно добавить элемент в цепочку/дерево внутри этого bucket


Это обеспечивает высокую пропускную способность: чтение ~бесплатное, запись масштабируется с числом потоков, без полной блокировки как в `Collections.synchronizedMap()`.

### TreeBin — красно-чёрное дерево при больших коллизиях

Если в одном бакете слишком много элементов, связный список превращается в **~={red}красно-чёрное дерево (TreeBin).=~**

Преимущества: 
- поиск и вставка становятся O(log n), а не O(n) 
- коллизии не превращаются в деградацию производительности

Это делает ConcurrentHashMap устойчивой к «плохим» ключам.

 
### Умный resize — без stop-the-world

Расширение таблицы (resize) — самое сложное место.

В обычном HashMap resize означает: 
1) блокируем всё 
2) переносим все элементы 
3) надеемся, что всё не рухнет 🤣. - ахахахахах бляяя

В ConcurrentHashMap:

✔ Resize выполняется постепенно

Бакеты переносятся порциями.

✔ Несколько потоков могут переносить параллельно

Они помогают друг другу.

✔ Нет глобальной блокировки

Чтения продолжаются свободно, пока идёт перенос.
