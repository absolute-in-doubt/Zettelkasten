Q: Расскажи про L1 cache в Hibernate.
A: L1 cache - кэш уровня EntityManager. Включён by default. Иногда его называют контекстом. 
	
При выполнении операций `EntityManager` типа `persist(..)`, `merge(..)` данные сохраняются именно в этот кеш. 
При выполнении `find(..)`, `findAll(...)` данные в первую очередь ищутся в L1  и если там нет (и не включён L2 cache) - запрос в БД.
При выполнении операции `remove(..)`, объект не удаляется напрямую из БД, а просто помечается как removed в кэше.
	
> [!warning]
> Bulk-UPDATE & bulk-DELETE выполняются напрямую в БД, минуя кеш.
	
При вызове `em.flush()` кэш gets flushed to the DB -> выполнябтся запросы. 
BTW: при коммите транзакции под капотом тоже вызывается `em.flush()`.
	
	
Данные в кэше хранятся в виде Key-Value пар: 
- Key (EntityClass, EntityId) 
- Value (экземпляр сущности)

Q: Расскажи про L2 cache в Hibernate.
A: L2 cache - кэш на уровне `EntityManagerFactory`.
	
> [!note] 
> Hibernate не поставляет реализацию L2 кэша из коробки — для этого нужно подключить любую из доступных реализаций. 
	
Алгоритм поиска сущностей:
```
1. Сначала ищет в L1 (быстро, локально)
2. Нет? → Ищет в L2 (медленнее, но все еще память)
3. Нет? → SQL к БД → сохраняет в L1 + L2
```
	
**Когда сущность попадает в L2** (при включенном L2):
	
1. **С `@Cacheable`** — всегда кэшируется в L2
    
2. **БЕЗ `@Cacheable`** — кэшируется **только при первом обращении** и если Hibernate считает это выгодным
	
---
**Пример:**
	
```java
// Транзакция 1
em.find(User.class, 1L); // L1 miss → L2 miss → БД → сохраняет в L1+L2
	
// В той же транзакции  
em.find(User.class, 1L); // L1 hit ✅ (супербыстро)
	
// Транзакция 2 (новый EntityManager)
em.find(User.class, 1L); // L1 miss → L2 hit ✅ (быстро, без БД)
```
	
---
**Note:**
	
Включая кэш второго уровня у вас пропадает возможность балансировать нагрузку (по крайней мере в лоб) между несколькими инстансами приложения, поскольку такие локальные кэши становятся несогласованными друг с другом. Есть у ehcache механизм репликации кэшей, но там свои минусы
	
И тем не менее, L2 кэш иногда используется — как правильно заметили выше, в основном для неизменяемых справочных сущностей

Q: Расскажи про query cache. Когда он применим?
A: Query cache используется для кэширования результатов запросов. Это эффективно, если у нас есть часто повторяющийся запрос, возвращающий небольшой объем данных. (Большие объемы данных держать в кэше нежелатьельно).
	
**Пример**:
	
```java
Query query = em.createQuery("SELECT u FROM User u WHERE u.status = :status");
query.setHint("org.hibernate.cacheable", true);
query.setParameter("status", Status.ACTIVE);
	
//или
	
Query query = em.createQuery("FROM MyEntity WHERE someProperty = :value") .setParameter("value", value)
.setCacheable(true);
```
