# @MockBean



**`@MockBean`** — аннотация из ~={red}**Spring Boot Test**=~, которая:

1. **Создаёт мок-объект** (как `@Mock` в Mockito).
2. **Добавляет его в Spring-контекст** (как бин), **переопределяя** реальный бин (если он есть).

---

## Отличия `@MockBean` от `@Mock + @InjectMocks`

|Критерий|@InjectMocks + @Mock|@MockBean|
|---|---|---|
|**Контекст**|Работает **вне Spring-контекста** (чистый Mockito)|Работает **внутри Spring-контекста**|
|**Переопределение бинов**|❌ Не подменяет реальные бины в Spring|✅ Подменяет реальные бины в Spring-контексте|
|**Требует Spring**|❌ Нет (используется в юнит-тестах)|✅ Да (используется в интеграционных тестах)|
|**Внедрение зависимостей**|Вручную через `@InjectMocks`|Автоматически в Spring-контексте|
|**Аннотации для теста**|`@ExtendWith(MockitoExtension.class)`|`@SpringBootTest`, `@WebMvcTest` и т.д.|
|**Пример использования**|Юнит-тесты изолированных классов|Интеграционные тесты с Spring|
|**Можно ли использовать с `@Autowired`**|❌ Нет|✅ Да|



---

## **📌 Почему нельзя всегда использовать `@Mock`?**

1. **`@Mock` не работает с Spring-контекстом**
    
    - Если тестируемый класс **зависит от Spring-бина** (например, `@Autowired`), то `@Mock` **не сможет его подменить**.
    - **Пример:**

```java
@Service
public class UserService {
    @Autowired
    private UserRepository userRepository; // Реальный бин
}
```
      
- ~={red}Если использовать `@Mock` для `UserRepository`, то в `UserService` **всё равно будет реальный бин**, а не мок.=~

2. **`@Mock` не переопределяет бины в контексте**
    
    - Spring **не знает** о моках, созданных через `@Mock`.
    - **Решение:** Использовать `@MockBean`, чтобы мок **заменил реальный бин** в контексте.
3. **`@Mock` требует ручного внедрения**
    
    - Для работы с `@Mock` нужно использовать `@InjectMocks` или вручную передавать моки в тестируемый класс.
    - **`@MockBean`** автоматически подменяет бин в **всем контексте**.

---

## Когда использовать `@MockBean`?

✅ **Интеграционные тесты** (с загрузкой Spring-контекста). 

✅ **Тестирование контроллеров** (`@WebMvcTest`), где нужно подменить сервисы/репозитории. 

✅ **Тестирование сервисов**, которые зависят от других Spring-бинов. 

✅ **Когда нужно переопределить реальный бин** в контексте.

---

## Когда использовать `@Mock`?

✅ **Юнит-тесты** (без Spring-контекста). 

✅ **Тестирование изолированных классов** (например, утилит, сервисов без зависимостей от Spring). 

✅ **Когда не нужно подменять бины в Spring-контексте**.

---

## Пример использования `@MockBean

```java
@SpringBootTest // или @WebMvcTest
public class UserControllerTest {

    @MockBean // Подменяет реальный UserRepository в контексте
    private UserRepository userRepository;

    @Autowired
    private UserController userController; // UserController получит мок-репозиторий

    @Test
    public void testGetUserById() {
        when(userRepository.findById(1L)).thenReturn(new User(1L, "Test"));
        User user = userController.getUserById(1L);
        assertEquals("Test", user.getName());
    }
}
```

- **Что происходит:**
    - `@MockBean` создаёт мок `UserRepository` и **заменяет реальный бин** в контексте.
    - `UserController` (внедрённый через `@Autowired`) **получает мок**, а не реальный репозиторий.

## Итоги

|Вопрос|Ответ|
|---|---|
|**Чем отличается `@MockBean` от `@Mock`?**|`@MockBean` работает с Spring-контекстом и переопределяет бины, а `@Mock` — только в Mockito.|
|**Почему нельзя всегда использовать `@Mock`?**|Потому что он не подменяет бины в Spring-контексте.|
|**Когда использовать `@MockBean`?**|В интеграционных тестах с Spring-контекстом.|
|**Когда использовать `@Mock`?**|В юнит-тестах без Spring.|
