# Стратегии кэширования

По сути 2 паттерна:

1. Cache Aside + обновление кэша приложением/Write Around
2. Read Through + Write Through/Back


---
### Cache Aside

![[Pasted image 20260401170155.png]]

- Это именно **Read strategy**
- Lazy-загрузка данных в кэш

**Плюсы:**
`+` Если кэш упадёт - приложение не сломается - просто будет читать данные из БД
`+` Подходит для read-heavy задач
`+` Схема кэша может отличаться от схемы БД (можем хоть DTO ответа хранить)

**Минусы:**
`-` Приложение может предоставлять устаревшие данные (если плохо настроен TTL)


---
### Read Through

Приложение читает из кэша, если в кэше нет, то **сам кэш читает из БД**.

![[Pasted image 20260401170749.png|502]]

- Это именно **Read strategy**
- Lazy-загрузка данных в кэш
- Приложение напрямую не обращается к БД ВООБЩЕ.

**Плюсы:**
`+` Упрощает код приложения


**Минусы:**
`-` Первое чтение - всегда cache miss.
`-` Схема БД и кэша должна быть одинаковой
`-` Приложение может предоставлять устаревшие данные (если плохо настроен TTL)


---
### Write Through

![[Pasted image 20260401171159.png]]

- Это именно **Write strategy**
- Приложение напрямую не обращается к БД. - Кэш сам пишет в БД.

**Плюсы:**
`+` Данные в кэше всегда актуальны
`+` Подходит, когда мало записи, но много чтения
`+` **~={red}Можно комбинировать с Read Through=~**

**Минусы:**
`-` Добавляет задержку на запись


---
### Write Back

Данные всегда пишутся в кэш, а БД обновляется асинхронно батчами.

![[Pasted image 20260401171610.png]]

- Это именно **Write strategy** - примерно то же самое, что и Write through но нет доп задержки при записи
- Приложение напрямую не обращается к БД. - Кэш сам пишет в БД.
- **Часто используемая реализация**

**Плюсы:**
`+` Данные в кэше всегда актуальны
`+` Подходит, когда много записи


**Минусы:**
`-` Если кэш упадёт, часть данных будет потеряна


---
### Write Around

Данные пишутся только в БД, а читаются только из кэша. **При cache miss приложени само ищет в БД и загружет данные в кэш.** -> **~={orange}Данные попадают в кэш только при чтении.=~**

![[Pasted image 20260401171941.png|452]]

- Это именно **Write strategy** - примерно то же самое, что и Write through но нет доп задержки при записи
- Приложение напрямую не обращается к БД. - Кэш сам пишет в БД.

**Плюсы:**
`+` Подходит для случаев, когда данные не запрашиваются сразу после записи (нужно время, пока кэш подтянет) 


**Минусы:**
`-` Приложение может предоставлять устаревшие данные (если плохо настроен TTL)
`-` Если читать сразу после записи - получим повышенную задержку
