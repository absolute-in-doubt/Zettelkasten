# Хранение данных - PostgreSQL


Данные каждой таблицы хранятся в отдельном файле или нескольких файлах. Внутри эти файлы делятся на страницы.

![[Pasted image 20260208195311.png]]

Каждая запись разделяется на 2 части:
- в начале файла хранятся ссылки на сами данные (оффсеты);
- с конца файла добавляются сами данные.

По мере записи движемся к середине:

![[Pasted image 20260208195446.png]]


##### Настройка размера оставляемого free space для каждого файла таблицы

По умолчанию `fill factor` = 1 -> по максимуму забиваем всё данными.

Если поставить `fill factor` = 0.5 -> только половина каждого файла будет заполнена данными. (Половина останется пустой).

***Для чего это надо?***
	Если мы меняем некий item (запись), то [[Многоверсионность и Конкурентность - PostgreSQL|он сам не меняется, а создаётся новая версия]]. Оставив место в файле, при изменении item-ов этого файла, новые версии сохранятся в free space.
	Если `fill factor` оставить = 1, то новая запись сохранится в конце какого-то (скорее всего другого) файла, который сейчас последовательно заполнятеся. => Для проверки видимости записи придётся два раза проверять их c random access алгоритмом -> медленнее.

### Вставляем невставляемое

Если вставляемый объект не влазит на страницу. (Ограничение по размеру для одной записи = 2Kb)

![[Pasted image 20260208200451.png]]

- сначала по классическому алгоритму запишет указатель в начале файла.
	- Postgres попытается сжать данные, чтобы размер < 2Kb. Сохранит с пометкой, что данные были сжаты. (При получении - обратно decompresses them)
	- Если не получилось - использует TOAST


###### TOAST - The Oversized Attribute Storage Technique

![[Pasted image 20260208200945.png]]

Item хранится по атрибутам (значениям колонок), ссылки хранятся в конце обычного файла таблицы. Части этого Item хранятся в TOAST слайсами: каждый изначальный кусок режется на части.

Header и Tail страниц в TOAST содержат информацию о том, как обратно сконструировать из кусков исходный Item при запросе пользователя.

Если запись типа:
```
( name, id, big_field)
```
То только big_field будет храниться в TOAST. И если при SELECT мы укажем `SELECT *`,  то файл будет сконструирован и отдан нам -> Если нам не нужны все поля, лучше явно указать, что нам надо, чтоб не ждать сборки тяжёлых атрибутов.
