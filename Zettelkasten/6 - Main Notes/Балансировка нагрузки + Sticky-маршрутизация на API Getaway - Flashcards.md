Q: Что произойдёт, если упадёт LoadBalancer?
A: LoadBalancer - single point of failure, если он упадёт - всеё приложение упадёт. Чтобы этого избежать можно:
- иметь несколько инстансов LoadBalancer-a
- health-checks & monitoring of LoadBalancers themselves - замечаем неполадки в работе заранее
- Auto-scaling & self-healing systems - как Controller Manager в Kubernetes (видит, что под упал (running 0/1) -> даёт команду поднять новый)
- В настройках DNS сервера указать IP запасного (Fail-over) LoadBalancer-а, куда перенаправить трафик, если сервер по основному IP упадёт

Q: Какие ты знаешь алгоритмы балансировки нагрузки?
A:       
**Базовые алгоритмы распределения**
	
- Round Robin (круговой)  
    Просто по очереди: запросы ходят 1‑й инстанс, 2‑й, 3‑й, затем снова 1‑й. Подходит для stateless‑сервисов, когда инстансы “примерно равны” по мощности и нагрузке.
    
- Weighted Round Robin  
    То же самое, но с весами: более мощный инстанс получает больше запросов (например, 60% / 30% / 10%). Нужен, когда инстансы различаются по ресурсам.
    
- Least Connections / Least Response Time  
    Новый запрос идёт на инстанс с наименьшим числом активных соединений или наименьшей задержкой ответа. Хорошо при неоднородной нагрузке (запросы “весят” по‑разному).
    
- Random  
    Каждый запрос уходит на случайный инстанс. Просто, иногда используется как резервный или в очень простых сценариях.
	
- **Geographical**
	Запрос отправляется на географически ближайший инстанс
	
	
---
**Sticky‑маршрутизация (липкие сессии)**
	
Sticky‑сессии нужны, если сервис **stateful**: важно, чтобы все запросы одного пользователя попали на один и тот же инстанс (пока он жив). Основные варианты:​
	
- **IP hash** 
    Хеш от IP клиента → выбор инстанса. Один и тот же IP всегда попадает на один и тот же инстанс (пока конфиг не меняется). Проблемы: NAT, смена IP, неравномерность распределения.​
    
- **Cookie‑based sticky**
    Балансировщик ставит специальную cookie и по ней решает, на какой инстанс дальше вести этого клиента. Более надёжно, чем IP hash, часто используется для веб‑сессий.​
    
- **Sticky по токену / session id**  
    Хеш от session id / user id → выбор инстанса. Это уже более “продвинутый” вариант, чаще в кастомных решениях.
    
	
**Важно:** 
	*sticky‑маршрутизация **маскирует проблему stateful‑состояния, но не решает её**. При падении инстанса сессии всё равно теряются, поэтому в современных системах обычно:*
	
- либо делают сервис stateless (авторизация, например),
    
- либо состояние выносят в общее хранилище и тогда sticky уже не так нужен.
