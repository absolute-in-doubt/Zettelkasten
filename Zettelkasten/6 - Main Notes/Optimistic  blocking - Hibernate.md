# Optimistic blocking - Hibernate


В Hibernate оптимистическая блокировка реализована на базе версионирования с использованием аннотации **~={purple}@Version=~** для поля класса (обычно, оно имеет тип`int`) .  ~={cyan}Это отдельное поле в таблице.=~

### Как это работает? 

Объект подгружается из базы данных, у него есть номер версии (если запись только создана - у нее номер версии 0). И когда изменения выгружаются, например, при вызове метода`entityManager.flush()`или при завершении транзакции (потому что здесь тоже вызывается метод`.flush()`), тогда текущий номер версии сравнивается с актуальной версией в базе данных . 

Если актуальная версия больше, чем текущая, то выбрасывается исключение`OptimisticLockException`, которое сигнализирует пользователю о том, что какой-то пользователь уже поменял объект и мы работаем с неактуальным состоянием. Если версия больше или равна той, которая в база данных, то запись фиксируется и версия увеличивается на один. 
Избежать исключения можно с помощью метода`entityManager.merge(...)`.

**Проще говоря, оптимистическая блокировка работает по принципу CAS (Compare And Swap): сравнил и, если версия больше, заменил.**


### Hibernate предоставляет два типа этой блокировки:
(дублирует [[LockModeType & PessimisticLockScope - JPA]])

1. `LockModeType.OPTIMISTIC`- блокировка по умолчанию для всех сущностей, помеченных аннотацией`@Version`.

2. `LockModeType.OPTIMISTIC_FORCE_INCREMENT`- блокировка полностью аналогичная прошлой, но проверяет версии дочерних элементов (например, связных с помощью отношений один ко многим).


Блокировки можно задавать с помощью следующих методов:
```java
entityManager.lock(...);

entityManager.find(..., LockModeType);

query.setLockMode(...);
```
