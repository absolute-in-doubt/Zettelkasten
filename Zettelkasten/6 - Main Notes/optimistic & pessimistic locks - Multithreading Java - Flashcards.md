Q: Что такое optimistic block?
A: Optimistic block - вид получения доступа к ресурсу без предварительной синхронизации, который предполагает, что ресурс скорее всего свободен. Поток читает его значение из Heap, проводит с ним операции, и только при записи нового значения сверяет, не было ли изменено изначальное значение. 
	
Если же изначальное значение было изменено -> кто-то параллельно редактировал ресурс -> операция текущего потока откатывается.
	
Преимущества:
1. отстутствие deadlocks & livelocks
2. высокий параллелизм
	
Минусы:
1. При частых конфликтах много откатов и повторов, возможна потеря «последнего» редактирования.
	
Пример:
- атомарные операции сравнения и установки (`compareAndSet`) в `Atomic*` классах, spincas-структуры, lock-free алгоритмы.
- StampedLock реализует возможность optimistick lock.
	
> Зачастую используется совместно с pessimistic lock: если не получилось - используем освобождения ресурса и захватываем pessimistic lock:
> ```java
> StampedLock lock = new StampedLock();
>
>// Поток A: optimistic read
>long stampA = lock.tryOptimisticRead();  // stamp = 1000
>double x = point.x; 
>double y = point.y;
>
>// Поток B: запись (меняет состояние!)
>long stampB = lock.writeLock();          // stamp = 1001 (счётчик> +1)
>point.x = 10; point.y = 20;
>lock.unlockWrite(stampB);
>
>// Поток A: проверка
>if (!lock.validate(stampA)) {            // 1000 != текущее состояние → false
>    // НЕ откатывает изменения B!
>    // Просто повторяет чтение под readLock
>    long newStamp = lock.readLock();
>    try {
>        x = point.x; 
>        y = point.y;        // теперь читает актуальные 10, 20
>    } finally {
>        lock.unlockRead(newStamp);
>    }
>}
> ```

Q: Что такое pessimistic block? Приведи примеры реализации в SQL и Java. Перечисли плюсы и минусы pessimistic block.
A: Это такой вид блокировки, который лочит запись до тех пор, пока не завершится транзакция. Пока запись удерживается, остальные пользователи ожидают, когда можно будет захватить запись.
	
- Исходит из предположения: конфликт «скорее всего будет».
    
- Поток/транзакция сначала захватывает блокировку ресурса, и пока она удерживается, другие не могут его менять (а иногда даже читать).
    
- Примеры в Java: `synchronized`, `ReentrantLock`, пессимистические lock-и в БД (`SELECT ... FOR UPDATE`, `PESSIMISTIC_WRITE` в JPA).
	
	
Плюсы:
- Сильные гарантии целостности: конфликтов при записи нет, потому что все конкуренты ждут.
	
Минусы:
- Меньше параллелизма, возможны очереди и дедлоки.
