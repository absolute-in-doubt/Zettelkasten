# Индексы - БД

Индекс — это структура данных (обычно B-дерево или хеш), которая ускоряет поиск строк в таблице без полного сканирования.

Преимущества

- ускорение поиска и сортировки по определенному полю или набору полей.
- обеспечение уникальности данных.

Недостатки

- требование дополнительного места на диске и в оперативной памяти и чем больше/длиннее ключ, тем больше размер индекса.
- замедление операций вставки, обновления и удаления записей, поскольку при этом приходится обновлять сами индексы.

- **Буферизация** - чтение и запись с помощью кэша.

#### Как достигается неизменяемость B-tree

Неизменяемость индексов обычно связана не с тем, что «индекс вообще никогда не меняется», а с MVCC‑подходом: существующая запись не перезаписывается, а появляются новые версии и/или новые узлы дерева.


Для INSERT и UPDATE примерно похожий алгоритм.

```sql
--изначально age = 30, id = 1

UPDATE users SET age = 31 WHERE id = 1;
```

Логика примерно такая:

1. В таблице не перезаписывается старая версия `(id=1, age=30)`, а создаётся **новая версия строки** `(id=1, age=31, txid=20)`, старая помечается как неактуальная для будущих транзакций.
    
2. В индексе:
    
    - Для ключа `30` со временем старая ссылочная запись будет считаться «указывающей на невидимую/устаревшую версию».
        
    - Для новой строки добавляется **новая запись**:
        
	```
	ключ: 31 → ссылка на версию строки (id=1, txid=20)
	```
        
    - **~={orange}Для старого ключа либо помечается, что версия больше не видна новым транзакциям (флаги видимости), либо запись позже физически вычищается вакуумом/GC.=~**
        

То есть «изменение» индекса при UPDATE — это:

- Добавить новую запись для нового значения ключа.
    
- Пометить старую запись как неактуальную или удалить её при обслуживании, а не править её «на месте».

##### Если узел B-tree переполнен

Когда узел переполнен:

- Индексный узел B‑tree делят на два (split): создаются новые страницы/узлы, ключи перераспределяются.
    
- В этом смысле «перестраивается узел дерева» — но и тут логика ближе к **созданию новых структур и переносу ссылок**, чем к точечному редактированию старой версии с точки зрения MVCC‑видимости.
