# ACID principles - БД


**A** - ***Atomicity** (атомарность)
**C** - ***Consistency*** (постоянство)
**I** - ***Isolation*** (изоляция)
**D** - ***Durability*** (надёжность)


### Atomicity

Система не может оказаться в некорректном промежуточном состоянии. 
Либо действие выполняется целиком, либо сохраняется начальное состояние.

Достигается транзакциями.

![[Pasted image 20251021075854.png]]


### Consistency

Ограничения, которые БД проверяет при произведении действий:
([[Column definition options (Constraints) - SQL|CONSTRAINT]] - CHECK и т.д., FOREIGN KEYs) - проверка корректности добавляемых данных.


![[Pasted image 20251021080048.png]]

Сейчас на БД не делегируют проверку правильности данных - это задача приложения. 

В микросервисной архитектуре консистентность, бывает, обеспечить невозможно, т.к. данные одной базы, принадлежащей одному сервису, могут быть связаны с данными драгой базы другого сервиса => валидируем через само приложение (сервисы).


### Durability

Сохранение данных на диск на случай падения БД. 
~={red}**Once committed, changes are permanent**=~

![[Pasted image 20251021080652.png]]

Изменения зачастую происходят в кэше. - БД сама определяет/мы задаём оптимальный способ to persist the data.

#### Уровни  изоляции

![[Pasted image 20251021083051.png]]

- **Read uncommited** - разрешено читать данные, изменённые, но ещё не зафиксированные другими транзакциями.
- **Read commited**  (реализуется через MVCC - версионирование изменений)- читаются только зафиксированные изменения. - ~={orange}**Postgres standard**=~
- **Repeatable read** - все данные, прочитанные транзакцией, гарантированно не изменятся другими транзакциями до её завершения. (**~={red}НЕ фиксит lost update#2 - на картинке не верно=~**)
- **Serializable** - полная изоляция: транзакции выполняются так, как будто они идут последовательно.

|Уровень|Что блокирует|Что НЕ блокирует|
|---|---|---|
|**REPEATABLE READ**|Блокирует **строки, которые уже читал** (row-level locks) + использует MVCC для версионности|Не блокирует **новые строки**, которые могут быть INSERT'нуты другими транзакциями в диапазон запроса|
|**SERIALIZABLE**|Блокирует **весь диапазон запроса** (range locks) + проверяет конфликты сериализуемости|—|


**~={orange}`SELECT FOR UPDATE/FOR SHARE` не привязаны к конкретному уровню изоляции и их использование необходимо для предотвращения lost update на всех уровнях изоляции кроме Serializable=~**

Их использование необходимо для предотвращения lost update именно на уровнях READ COMMITTED и REPEATABLE READ, где без блокировок MVCC позволяет двум транзакциям прочитать одну строку, обновить её на основе старых данных и перезаписать изменения друг друга. В SERIALIZABLE lost update второго типа предотвращается автоматически за счёт проверки сериализуемости (транзакция может быть отменена при конфликте), так что явные блокировки не обязательны, хотя и усиливают контроль.

> [!warning]
> Операции `UPDATE`, `DELETE` блокируют ту строку, на которой выполнялись до конца транзакции:
> ```sql
> BEGIN;
>SELECT balance FROM users WHERE id=1;  -- Локов НЕТ. Просто читаешь.
>-- тут 5 секунд думаешь в коде...
>UPDATE users SET balance = ... WHERE id=1;  -- ВОТ ТУТ взялся лок
>-- лок держится до COMMIT
>COMMIT; -- вот тут лок отпустился
> ```
> Это предотвращает Lost update I, но не предотвращает Lost Update 2-го типа.
