### Архитектура репликации


Пишем в мастер, синхронные реплики - для воссстановления после падения, асинхронные - только для аналитических запросов.

Default cluster (по одной):
- Master - всегда один. (Обычно и пишем сюда и читаем отсюда. Пишем в Sync ток если прям оч надо размазать нагрузку).
- Sync - ставим больше одной только, если очень много чтений для ускорения чтений
- Async - для аналитиков. Отстаёт о

#####  **Master-slave**

Одна ведущая реплика. В одну пишем, читать можем из многих.

![[Pasted image 20260213201700.png]]
 
- Операторная репликация - пересылает команды follower-ам
  `-` проблема с триггерами, транзакциями успешна в одной базе, некспешна в другой. В одной базе позитивная блокировка сработала, в другой нет -> неконсистентность.

- Физическая (перенос с [[Диспетчер журналов (WAL) - PostgreSQL|WAL log]]) - журнал в который пишутся все изменения. Если транзакция не прошла - туда не пишется ничего. Из него пересылаются все изменения на slaves.

- Логическая - физическая репликация универсальным языком с обработкой исходных сообщений. Можно добавить логику. (Есть модуль, трансформирующий данные). Можно передавать данные между разными БД.

- Триггерная репликация - Мы отвечаем за репликацию на уровне приложения (пишем данные сразу во все БД ,  или одни данные в одну, дргуе в другую).




Change Data Capture - захватываем сообщения 

- **Master-master** - Несколько In-Sync replicas - (пример миграция с одной БД на другую)

### Поводы для репликаци

- **надежность** (отказоустойчивость) - дублируем данные на случай падения одной из инстансов БД.

- **распределение нагрузки** (Масштабируемость)- на больших базах одна база может не вытаскивать всю нагрузку и иногда имеет смысл добавить реплику для размазывания нагрузки
  + Postgres не любит много конннектов - решается пулом, но не всегда.


- Снижение latency за счет распределения для разных ГЕО.
