# JVM multithreading issues - Java


## Instruction reordering

Это когда компилятор, JIT или CPU **меняют порядок выполнения инструкций**, если с точки зрения одного потока результат не меняется.  
Проблема появляется в многопоточности: другой поток может увидеть **частично опубликованное состояние**. Поэтому нужны `volatile`, `synchronized`, `Lock`, happens-before и другие барьеры видимости.

Короткая формулировка:

- **В одном потоке** порядок может быть “оптимизирован”.
    
- **Между потоками** без синхронизации этот порядок нельзя считать гарантированным.

Простой пример — **double-checked locking без `volatile`/synchronized**: из-за reordering другой поток может увидеть объект как “уже созданный”, хотя его поля еще не инициализированы

```java
class Singleton {
    private static Singleton instance;

    static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

#### Что может пойти не так

Создание объекта логически состоит из нескольких шагов:

1. Выделить память.
    
2. Инициализировать объект.
    
3. Присвоить ссылку `instance`.

Без синхронизации/`volatile` порядок этих шагов может быть переупорядочен, и другой поток увидит `instance != null` раньше, чем объект реально полностью готов.

## Spurious wakeup

Это когда поток просыпается из `wait()` или `await()` **без явного `notify()/signal()`**.  
Это разрешено спецификацией, поэтому ждать нужно **в цикле**, а не через `if`.

**~={red}?!=~** Это может произойти не только с `Object.wait()`: **может быть и с `ReentrantLock` тоже**, и с другими объектами синхронизации.


Пример формулировки для собеса:

- `wait/await` всегда надо вызывать в `while`, где повторно проверяется условие.
    
- Проснувшийся поток не должен сразу предполагать, что нужное состояние уже наступило.

```java
synchronized (lock) {
    while (!condition) {
        lock.wait();
    }
}
```


### Одной фразой

**Reordering ломает видимость порядка действий между потоками, а spurious wakeup требует проверять условие ожидания в цикле после каждого пробуждения.**
