Q: Расскажи про checked & unchecked exceptions.
A:    
	
| Аспект                  | Checked Exceptions                                   | Unchecked Exceptions                                       |
| ----------------------- | ---------------------------------------------------- | ---------------------------------------------------------- |
| **Наследование**        | Подклассы `Exception` (не `RuntimeException`)        | Подклассы `RuntimeException` или `Error`                   |
| **Проверка**            | Компилятор требует обработку (try-catch или throws)​ | Не проверяются компилятором ​                              |
| **Время возникновения** | Compile-time (предсказуемые внешние факторы)         | Runtime (ошибки программирования) ​                        |
| **Примеры**             | `IOException`, `SQLException`                        | `NullPointerException`, `ArrayIndexOutOfBoundsException` ​ |
| **Обработка**           | Обязательна для вызова API                           | Опциональна (рекомендуется избегать) ​                     |
	
---
**Когда использовать**
	
- **Checked**: Внешние факторы вне контроля программы (файлы, сеть, БД). Позволяют восстановиться: "файл не найден — создайте новый".​
    
- **Unchecked**: Баги в коде (деление на 0, null-доступ). Их цель — обнаружить и исправить во время разработки, а не обрабатывать в продакшене

Q: Обязан ли абстрактный класс явно реализовывать методы интерфейса, который он реализует? (В виде abstract methods или в виде методов с реализацией)
A:    
Абстрактный класс, реализующий интерфейс (`implements`), **может оставить методы интерфейса абстрактными** — без реализации. Обязанность реализовать все методы переходит к **конкретным (неабстрактным) подклассам**
	
---
**Пример:**
	
```java
interface Flyable {
    void fly();  // Метод интерфейса
}
	
abstract class Bird implements Flyable {
    // НЕ реализует fly() — оставляет абстрактным
    // Компилятор не ругается!
}
	
class Eagle extends Bird {
    @Override
    public void fly() {  // ТОЛЬКО здесь обязательна реализация
        System.out.println("Eagle flies high");
    }
}
```
	
**Сравнение обязанностей**
	
| Тип класса            | implements интерфейс   | Обязан реализовать ВСЕ методы?        |
| --------------------- | ---------------------- | ------------------------------------- |
| **Обычный класс**     | ✅ `implements Flyable` | **ДА** — все методы                   |
| **Абстрактный класс** | ✅ `implements Flyable` | **НЕТ** — может оставить абстрактными |
	
**Почему так работает**
	
- Интерфейс = **контракт** (декларации методов).
    
- Абстрактный класс = **частичный шаблон** (может реализовать часть контракта).
    
- Только **финальные классы** полностью выполняют контракт.

Q: Чем отличаются StringBuilder и StringBuffer?
A: StringBuilder и StringBuffer в Java — это изменяемые классы для эффективной работы со строками, в отличие от неизменяемого String. Главное отличие: StringBuffer потокобезопасен (синхронизирован), а StringBuilder — нет и быстрее.
	
---
**Потокобезопасность**
	
StringBuffer использует синхронизацию методов (synchronized), что предотвращает конфликты при многопоточном доступе, но снижает производительность.  
StringBuilder не синхронизирован, поэтому подходит для однопоточных приложений и работает быстрее.
	
---
**Производительность**
	
StringBuilder обычно в 1,5–2 раза быстрее StringBuffer из-за отсутствия накладных расходов на синхронизацию.  
Оба класса имеют одинаковый API (append, insert, delete и т.д.) и используются для конкатенации строк в циклах.

Q: Чем отличаются `git reset` и `git revert`?
A: reset переписывает историю коммитов, а revert сохраняет.
	
---
**Git Reset**
	
Перемещает указатель ветки (HEAD) на указанный коммит, удаляя или изменяя историю после него.  
Доступны режимы:
	
| Команда                      | HEAD           | Staging Area                                 | Working Directory |
| ---------------------------- | -------------- | -------------------------------------------- | ----------------- |
| `git reset --soft <commit>`  | ← перемещается | **НЕ трогается** (изменения остаются staged) | НЕ трогается      |
| `git reset --mixed <commit>` | ← перемещается | **Сбрасывается** (unstage всех изменений)    | НЕ трогается      |
| `git reset --hard <commit>`  | ← перемещается | Сбрасывается                                 | **Сбрасывается**  |
	
---
**Git Revert**
	
Создаёт новый коммит, который отменяет изменения указанного коммита, не трогая историю.  
Безопасен для командной работы и опубликованных веток, так как история остаётся полной и линейной.  
Идеален для исправления ошибок в уже запушенных коммитах.

Q: Какими принципами разработки ты руководствуешься? Какие вообще знаешь? Какой принцип цаще всего нарушается? Почему? Приведи пример?

Q:  Для чего используется epsilon gc?
A:     
-  Это экспериментальный сборщик мусора, введённый в Java 11, который не освобождает память, а только выделяет её, пока не возникнет OutOfMemoryError. Он обеспечивает минимальный overhead на GC, полностью исключая паузы от сборки мусора.​
	
**Основные применения**
	
- Тестирование производительности: позволяет измерить скорость приложения без влияния GC.​
    
- Оценка memory pressure: показывает объём выделяемой памяти без очистки.
    
- Короткоживущие задачи: идеален для программ, завершающихся до исчерпания heap (например, скрипты).
	
- `-XX:+UseEpsilonGC`.

Q: Чем @Controller и @Repository отличаются от @Component?
A:   
**@Controller**
	
Отвечает за обработку HTTP-запросов в веб-слое (Spring MVC).
	
- Распознаётся `DispatcherServlet` для маппинга методов с `@RequestMapping`.
    
- Позволяет использовать `@ResponseBody`, `@RestController` (комбинация с `@Controller` + `@ResponseBody`).
    
- Семантически указывает на контроллер-презентер, улучшая читаемость и инструменты IDE.
    
---
**@Repository**
	
Используется для DAO/репозиториев в персистентном слое.
	
- Автоматически переводит исключения платформы (JDBC, JPA) в Spring-исключения (`DataAccessException`).
    
- Добавляет AOP-аспект для обработки ошибок БД без boilerplate-кода.
    
- Поддерживает транзакции и интеграцию с `JdbcTemplate`, JPA
	
| Аннотация     | Слой        | Доп. функциональность    |
| ------------- | ----------- | ------------------------ |
| `@Component`  | Любой       | Базовая: авто-детект, DI |
| `@Controller` | Web         | Маппинг запросов         |
| `@Repository` | Persistence | Обработка исключений БД  |

**Pattern matching for instanceof** — расширение оператора `instanceof`, которое **автоматически создает и приводит переменную** при успешной проверке типа. **НЕ новый оператор**, а **улучшение синтаксиса instanceof**.
	
**До Java 16 (классика)**
	
```java
if (obj instanceof String) {
    String s = (String) obj;  // ❌ Ручной cast
    System.out.println(s.length());
}
```
	
**Java 16+ (Pattern Matching)**
	
```java
if (obj instanceof String s) {  // ✅ Авто-приведение
    System.out.println(s.length());
}
```

Q: Чем использование JdbcTemplate лучше прямого использования JDBC?
A: **JdbcTemplate значительно лучше прямого JDBC** за счет автоматизации boilerplate-кода и устранения типичных ошибок.
	
**Основные преимущества**
	
|Проблема в чистом JDBC|Решение JdbcTemplate|
|---|---|
|`Connection conn = DriverManager.getConnection()`|Авто-получение из DataSource|
|`try-finally` для `close()` каждого ресурса|**Автоматическое управление ресурсами** (try-with-resources внутри)|
|`SQLException` (проверяемое)|**RuntimeException** (`DataAccessException`) с детализацией|
|Ручной парсинг `ResultSet`|`query().map(row -> new User(...))`|
|Повторяющийся код для каждого запроса|Один метод на запрос|
	
---
**Что НЕ оптимизирует JdbcTemplate**
	
- Не переписывает ваш SQL (нет батчинга, переиндексации)
    
- Не анализирует планы запросов
    
- Не применяет кэширование на уровне SQL
    
- **Производительность = чистому JDBC** (миллисекунды не потеряны)

Q: Можно ли в Java при наследовании сужать область возвращаемого типа метода?
A: Да, **можно** — но только **для возвращаемого типа**, и только если новый тип является **подтипом** исходного. Это называется **ковариантный возвращаемый тип**.[](https://sobes.tech/bank/java/7a079c00-2ad7-4578-bba8-48278c61f638)
	
Например:
	
```java
class Animal {}
class Dog extends Animal {}
	
class Parent {
    Animal get() { return new Animal(); }
}
	
class Child extends Parent {
    @Override
    Dog get() { return new Dog(); }
}
```
	
**?! Важный нюанс**
	
Сужать можно **тип результата**, но **нельзя сужать видимость метода** при переопределении: `public` нельзя заменить на `protected` или `private`.

Q: Расскажи как работает git и в чем его приемущества по сравнению с аналогами.
A: **Git под капотом:**
	
- Git хранит не изменения строк, а **снимки (snapshots)** всего проекта.
- Каждый коммит — это объект с:
    - ссылкой на корневое дерево файлов (tree),
    - автором,
    - сообщением,
    - ссылками на родительские коммиты.
        
- Все объекты (blob, tree, commit, tag) адресуются по **SHA-хешу**.
- Ветки — это просто **указатели на коммит**. Переключение ветки = перемещение указателя.
- История образует **направленный ациклический граф (DAG)** коммитов.
    
**Почему Git быстрый?**
	
- Большинство операций работает только с локальной БД `.git`.
- Не нужно обращаться к серверу для просмотра истории, создания веток, коммитов и слияний.
- Ветка занимает буквально несколько байт (ссылка на коммит).
    
**Почему лучше многих аналогов (SVN, CVS)?**
	
- Полная история у каждого разработчика.
- Нет единой точки отказа.
- Дешёвые ветки и быстрые merge.
- Целостность данных гарантируется хешами: подменить коммит незаметно нельзя.
- Объектная модель (commit/tree/blob) проще и гибче централизованных систем.
    
**Запомнить:**
	
> Git — это локальная база данных объектов, связанных хешами в DAG; ветка — просто указатель на коммит. Именно поэтому Git быстрый и удобный для параллельной разработки.

Q: Чем отличаются git pull от git fetch?
A: **`git fetch`**
	
- Скачивает изменения с удалённого репозитория.
    
- **Не меняет** текущую ветку и файлы.
    
- Только обновляет удалённые ссылки (`origin/main` и т.д.).
    
	
**`git pull`**
	
- Делает `fetch` + сразу интегрирует изменения в текущую ветку (`merge` или `rebase`).
    
- Может изменить рабочие файлы и вызвать конфликты.
    
	
> `fetch` — «посмотреть, что изменилось».  
> `pull` — «скачать и применить изменения».

Q: расскажи про git flow.
A: **Git Flow** — стратегия работы с ветками в Git.
	
**Основные ветки**
	
- **main (master)** — стабильный код в продакшене.
    
- **develop** — основная ветка разработки.
    
**Вспомогательные ветки**
	
- **feature/*** — разработка новой функциональности.
    
    - Создаётся от `develop`.
        
    - После завершения вливается обратно в `develop`.
        
- **release/*** — подготовка релиза.
    
    - Создаётся от `develop`.
        
    - После тестирования вливается в `main` и `develop`.
        
- **hotfix/*** — срочное исправление продакшена.
    
    - Создаётся от `main`.
        
    - После исправления вливается в `main` и `develop`.
        
	
**Схема**
	
```text
main
 ├── hotfix/*
 │
develop
 ├── feature/*
 ├── feature/*
 └── release/*
```
	
**Плюсы**
	
- Чёткое разделение разработки и продакшена.
    
- Удобно для больших команд и редких релизов.
    
**Минусы**
	
- Много веток и merge'ей.
    
- Избыточен для небольших проектов и CI/CD.
    
**Запомнить:**
	
> `main` — прод, `develop` — разработка, `feature` — фича, `release` — релиз, `hotfix` — срочный фикс.

Q: что такое boxing и unboxing?
A: **Boxing (autoboxing)** — автоматическое преобразование примитива в объект-обёртку.
	
```java
int x = 10;
Integer y = x; // boxing
```
	
**Unboxing** — автоматическое преобразование объекта-обёртки в примитив.
	
```java
Integer y = 10;
int x = y; // unboxing
```
	
**Зачем нужны обёртки?**
	
- Дженерики работают только с объектами:
    
```java
List<Integer> list = new ArrayList<>();
```
	
- Можно хранить `null`.
    
**Опасность:**
	
```java
Integer x = null;
int y = x; // NullPointerException
```
	
**Запомнить:**
	
> Boxing: `int → Integer`  
> Unboxing: `Integer → int`  
> Выполняются автоматически компилятором.
	
**Примеры:**
	
**1. Добавление примитивов в коллекции**
	
```java
List<Integer> numbers = new ArrayList<>();numbers.add(10); // autoboxing: int -> Integer
```
	
Самый частый случай.
	
---
**2. Получение значения из коллекции**
	
```java
Integer value = numbers.get(0);int x = value; // unboxing
```
	
---
	
**3. Сравнение с примитивом**
	
```java
Integer x = 100;if (x == 100) { // unboxing    ...}
```
	
Компилятор преобразует `x` в `int`.
	
---
**4. Арифметические операции**
	
```java
Integer a = 10;Integer b = 20;int sum = a + b; // оба unboxing
```
	
На самом деле:
	
```java
int sum = a.intValue() + b.intValue();
```
	
---
**5. Возврат из методов Stream API**
	
```java
List<Integer> list = List.of(1, 2, 3);int sum = list.stream()              .mapToInt(Integer::intValue).sum();
```
