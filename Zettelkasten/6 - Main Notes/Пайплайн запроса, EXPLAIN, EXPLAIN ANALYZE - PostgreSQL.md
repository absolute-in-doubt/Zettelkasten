# Пайплайн запроса - PostgreSQL


### Parser

Парсит SQL запрос на части.


---
### Analyzer

Строит более детализированное дерево запроса. 
Проверяет существуют ли такие таблицы, поля. Проверяет права доступа пользователя


---
### Planner 

На основе распаршенного дерева планирует наиболее эффективный запрос



---
###### EXPLAIN 

![[Pasted image 20260215160333.png]]

`EXPLAIN` проводит 3 этапа и возвращает план, но не выполняет последний этап (Executor).

cost - цена (абстрактные единицы), на основе, которой Planner определяет оптимальный вариант.

rows - кол-во строк, ожидаемое в результате

width - в байтах размер строки (одной записи). **Большая width - плохо, долго вычитываем из файла.**



----
### Executor

Исполняет запрос

![[Pasted image 20260215155402.png]]

work_mem - оперативная память, где работают backend процессы (очень маленькое by default)

Database - общее дисковое хранилище внутри БД.



###### EXPLAIN ANALYZE

`EXPLAIN ANALYZE` проводит все 4 этапа

**[[Организация доступа к данным - PostgreSQL|Seq scan]]** - тип организации доступа к данным
![[Pasted image 20260215160535.png]]

Результат именно в милисекундах.

rows сверху - кол-во строк, ожидаемое в результате
rows снизу - кол-во строк реально возвращённых

loops - кол-во итераций по таблице.
Rows Removed By Filter - кол-во строк отброшенных фильтром. Должно быть значительно меньше кол-ва возвращённых.

Buffers - кол-во данных, которе прочитали из буфера (**~={pink}БД кэширует данные=~**)
- shared hit - читаем с диска, не с оперативы (одна единица - одна страница)

Planning time - время построения плана запроса. Если первые 3 стадии кэшировать (**~={purple}Prepare && Execute=~**) - его не будет.
