# Перенаправление запросов между сервлетами

- В Jakarta Servlet существует два основных механизма «передачи» обработки между сервлетами: **server‑side forward/include** (через `RequestDispatcher`) и **client‑side redirect** (через методы ответа `sendRedirect`). 

### Server‑side перенаправление (forward/include)

- `RequestDispatcher` позволяет внутри сервера передать управление другому сервлету или JSP без нового HTTP‑запроса со стороны клиента:  
  - `forward(request, response)` — полностью передаёт обработку другому ресурсу.  
  - `include(request, response)` — «включает» вывод другого ресурса в текущий ответ.
  
- Клиент при `forward` не видит изменения URL, так как перенаправление происходит внутри контейнера, и запрос/ответ остаются теми же объектами (`HttpServletRequest` / `HttpServletResponse`).

Пример использования `RequestDispatcher` в сервлете:

```java
String target = "/nextServlet";
RequestDispatcher dispatcher = request.getRequestDispatcher(target);
dispatcher.forward(request, response);
```


## Client‑side перенаправление (sendRedirect)

- Метод `response.sendRedirect("url")` отправляет клиенту ответ с кодом 3xx (обычно 302) и заголовком `Location`, после чего браузер сам инициирует новый HTTP‑запрос по указанному адресу. 
    
- При `sendRedirect`:
    
    - Меняется URL в адресной строке клиента.
        
    - Создаётся новый запрос/ответ, поэтому параметры нужно передавать в query‑строке или через другие механизмы (cookies, сессия). 
        

Пример `sendRedirect` в сервлете:

```java
response.sendRedirect(request.getContextPath() + "/otherServlet");
```

## Выбор подхода

- Использовать **forward** целесообразно, когда нужно внутренне делегировать обработку (например, контроллер → JSP), сохраняя текущий запрос и атрибуты. 
    
- Использовать **redirect** удобно при Post/Redirect/Get (PRG) или при переходе на другой контекст/домен, когда требуется новый запрос и явное изменение URL.
