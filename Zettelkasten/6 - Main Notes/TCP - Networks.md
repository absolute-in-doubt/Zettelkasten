# TCP - Networks

TCP - Transmission Control Protocol.

**~={orange}TCP - Connection oriented protocol=~:** 
	он в первую очередь устанавливает сессию между двумя общающимися компьютерами.

TCP гарантирует, что:

- все данные будут доставлены

- данные будут доставлены в нужном порядке


> [!example] Note
> После установления TCP-соединения  **обе стороны могут одновременно отправлять и получать данные**, это называется **полный дуплекс (full-duplex)**.


### Установление TCP соединения

![[Pasted image 20260626203701.png]]

1. Отправитель шлёт SYN-сообщение

2. Receiving side sends back  SYN ACK - acknowledgement message: оповещает отправителя, что он получил сообщение

3. Отправитель шлёт ACK RECEIVED сообщение получающей стороне


После этого отправитель начинает отправлять данные.


### Отправление данных (после установки соединения)


TCP гарантирует доставку данных за счёт комбинации нескольких механизмов:

1. **Нумерация байтов (Sequence Numbers)**  
    Каждый байт в TCP-потоке имеет свой номер. Благодаря этому получатель может:
    - определить, какие данные уже получены;
    - восстановить правильный порядок данных;
    - обнаружить пропуски.
	
	В заголовке каждого TCP-сегмента есть **Sequence Number** — номер **первого байта** данных в этом сегменте.
	
	Например:
	
	```
	Сегмент 1:
	Sequence Number = 1
	Data = байты 1-5
	
	Сегмент 2:
	Sequence Number = 6
	Data = байты 6-11
	```

2. Проверка checksum:
	
![[Pasted image 20260628134242.png]]
	
Отправитель делает checksum сегмента и добавляет к сообщению. Получатель тоже делает checksum и сравнивает с checksum сообщения.
	
Если checksum не совпадает - получатель не отправляет ACK.


3. **Подтверждения получения (ACK)**  
    Получатель действительно отправляет подтверждения, но **не обязательно после каждого пакета**.
    
    Например:
    
    ```
    Клиент -> [байты 1-1000]
    Клиент -> [байты 1001-2000]
    
    Сервер -> ACK 2001
    ```
    
    ACK `2001` означает:
    
    > «Я успешно получил все байты до 2000 включительно и теперь ожидаю байт 2001».
    
    TCP использует **кумулятивные ACK**: одно подтверждение может подтвердить получение сразу нескольких сегментов.




4. **Повторная передача (Retransmission)**  
    Если отправитель не получил ACK за определённое время (**Retransmission Timeout, RTO**), он считает сегмент потерянным и отправляет его повторно.


5. **Fast Retransmit**  
    Если отправитель получает несколько одинаковых ACK подряд (обычно 3 duplicate ACK), он может не ждать таймаута и сразу переотправить потерянный сегмент.


6. **Контроль потока (Flow Control)**  
    Получатель сообщает, сколько данных он готов принять (Receive Window), чтобы отправитель не переполнил его буфер.



### Закрытие TCP соединения


```
Клиент                              Сервер

FIN ------------------------------>
       "Я больше не буду отправлять данные"
	
     <----------------------------- ACK
       "Ок, я понял"
	
     <----------------------------- FIN
       "Я тоже закончил отправку"
	
ACK ------------------------------>
       "Ок, я понял"
```
