Q: Когда включается Minor GC и когда Full GC в Serial GC?
A:   
	
| Тип GC       | Когда запускается                                                        | Область                        | Stop-the-world                |
| ------------ | ------------------------------------------------------------------------ | ------------------------------ | ----------------------------- |
| **Minor GC** | Eden заполнен (нет места для нового объекта)                             | Только Young (Eden + Survivor) | Да, весь хип не затрагивается |
| **Full GC**  | Tenured (Old) заполнен **ИЛИ** Survivor не может принять объекты из Eden | Вся куча (Young + Old)         | Да, полная остановка.         |

Q: Когда включается Minor GC и когда Mixed GC в G1 GC?
A:     
**G1 GC** (Garbage First) — generational low-pause collector (с Java 7, по умолчанию с Java 9+). Разбивает heap на **регионы** (1–32 МБ), собирает **young** (Eden + Survivors) и **old** по мере "грязности" (по liveness, RSet), стремясь к паузам ≤ `MaxGCPauseMillis` (200 мс по умолчанию)
	
- `-XX:+UseG1GC`.​
	
Фазы G1 GC: 
	
1. Minor GC + Initial Markup - копирующий GC, выполняется при заполенении памяти до определенного порога очистка young generation регионов. Запускается при заполнении Eden (~50% young). Определяет **Initial Mark** (корни: потоки, JNI, статические поля).
   ~={orange}Young GC **включает** Initial Mark (STW), но это не "Minor GC +", а единая пауза=~
	
2. **Root Region Scan**: Конкурентно сканирует Survivor-регионы на ссылки в old (для Concurrent Mark). Завершается перед следующим Young GC.
    
3. **Concurrent Mark**: Конкурентно (параллельно приложению) строит **liveness** регионов (SATB-алгоритм). Прерывается Young GC. Завершается **Remark** (STW: сливает SATB-батчи, финальная пометка).
    
4. **Cleanup**: STW/конкурентно: подсчёт свободного места, возврат пустых регионов в free list.
    
5. **Mixed GC**: После Mark (если old заполнен > `-XX:InitiatingHeapOccupancyPercent=45%`): STW evacuation **old + young** регионов (по "грязности"). До Full GC.
    
	
**Full GC**: STW, если mixed не справляется (редко).
1. Remark + Mixed GC - после завершения Concurrent Markup - происходит STW пауза, завершается пометка живых объектов (Remark) и запускается копирующий коллектор, который копирует из наиболее грязных регионов всех типов в более чисты регионы следующего типа, например из S0 в S1, из S1 в Old. Из самых грязные Old generation regions данные копируются в более чистые регионы и очищенные регионы превращаются в Eden regions
	
**Параметры**: `-XX:MaxGCPauseMillis`, `-XX:G1HeapRegionSize`, `-XX:InitiatingHeapOccupancyPercent`.
	
| Тип GC       | Триггер запуска                                                                                 | Области сборки                                       |
| ------------ | ----------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| **Minor GC** | Eden заполнен, нет места для новых объектов                                                     | Только Young регионы (Eden + Survivor).              |
| **Mixed GC** | Concurrent marking завершён **И** Old Gen > `InitiatingHeapOccupancyPercent` (45% по умолчанию) | Young + N Old регионов с большим количеством мусора. |

Q: Какая JVM по умолчанию идет в JDK? И какой в ней используется GC по умолчанию?
A: **В JDK по умолчанию используется HotSpot JVM от Oracle/OpenJDK.**  
**GC по умолчанию**: G1 (с JDK 9+), в JDK 8 — ParallelGC.

Q: Что вы знаете про Epsilon GC? Когда он используется? Приведите примеры.
A: Epsilon GC в Java — это экспериментальный сборщик мусора, введённый в Java 11, который не освобождает память, а только выделяет её, пока не возникнет OutOfMemoryError. Он обеспечивает минимальный overhead на GC, полностью исключая паузы от сборки мусора.
	
**Основные применения**
	
- Тестирование производительности: позволяет измерить скорость приложения без влияния GC.
    
- Оценка memory pressure: показывает объём выделяемой памяти без очистки.
    
- Короткоживущие задачи: идеален для программ, завершающихся до исчерпания heap (например, скрипты).
