Q: Какие есть способы организации доступа к данным в Postgres?
A:    
Sequential scan:
`+` Хорошо, когда читаем почти все данные + OS кэширует данные
`+` Если табличка маленька 10-100 записей - она уместиться в 1-2 страницы -> sequential scan будет самым эффективным. Какие бы индексы не добавляли - всё равно на `EXPLAIN`,  ``EXPLAIN ANALYZE`` получим sequential scan.
`-` Хуйня, если надо прочитать небольшую подвыборку
	
---
Index Scan:
`+` Диапазон значений
`+` Маленькая выборка (<5% таблицы)
`-` хаотичное чтение таблиц
`-` средние и большие выборки читаются медленно
	
---
Bitmap Heap Scan
`+` идём в Heap единожды для каждого элемента
`+` для запросов, возвращающих >10% строк
	
Строится битовая карта - промежуточная структура. Заполняем её за один проход. 
	
Первый проход - заполняем битовую маску. (со всем данными, и устаревшими и нет, игнорируя те, что отмечены как `LP_DEAD`)
	
Когда составили карту - просматриваем данные из heap.
	
---
Index Only Scan - **самый оптимальный способ**
	
Все данные, что мы ищем, просматриваем через карту видимости. Если карта видимости подтверждает, что данные живые - не ходим в кучу искать актуальную версию. Если данные dead - ищем в heap актуальную версию.
	
![[Pasted image 20260214180138.png]]
	
Если найденный указатель указывает на страницу, которая в карте видимости указана как зелёная - возвращаем значение из кэша (буфера), если же красная - приходится идти в heap и проверять (тут только один раз пришлось)
	
> [!tip] Note
> Чтобы получение данных работало через Index Only Scan можно использовать:
> -  частичные индексы (например индекс по задачам, у которых статус "ToDo", т.к. они скорее всего будут alive):
> ```sql
> CREATE INDEX idx_name ON table(column) WHERE condition;  -- напр. WHERE status = 'active'
> ```
> - функциональные индексы:
> ```sql
> CREATE INDEX idx_lower_email ON users(LOWER(email));
> ```
> Планировщик выбирает Index Only Scan, если индекс содержит все запрашиваемые столбцы (включая `ctid` для MVCC) и `WHERE` совпадает с предикатом — без доступа к heap таблицы. Частичные/функциональные индексы подходят, т.к. плотно покрывают данные запроса.

Q: В каких случаях будет использоваться Index Only Scan?
A:    
Чтобы получение данных работало через Index Only Scan можно использовать:
-  частичные индексы (например индекс по задачам, у которых статус "ToDo", т.к. они скорее всего будут alive):
```sql
CREATE INDEX idx_name ON table(column) WHERE condition;  -- напр. WHERE status = 'active'
```
	
- функциональные индексы:
```sql
CREATE INDEX idx_lower_email ON users(LOWER(email));
```
	
Планировщик выбирает Index Only Scan, если индекс содержит все запрашиваемые столбцы (включая `ctid` для MVCC) и `WHERE` совпадает с предикатом — без доступа к heap таблицы. Частичные/функциональные индексы подходят, т.к. плотно покрывают данные запроса.
	
То есть для запроса без параметров будет выбран Index Only Scan по обычному индексу.

Q: В каких случаях будет применён Bitmap Scan для организации доступа к данным?
A: `Bitmap Heap Scan` применяется, когда нужно **выбрать много строк через индекс, но прочитать только нужные страницы кучи последовательно**, вместо `Seq Scan` или частого рандомного чтения по `Index Scan`.
	
**Основные случаи применения**
	
1. **Большой результат по индексу**
    
    - Запрос выбирает **много строк** (например, `WHERE total_amount < 10000`), но для этого есть индекс.
        
    - PostgreSQL сначала делает `Bitmap Index Scan` (формирует битовую карту страниц), затем `Bitmap Heap Scan` — читает только помеченные страницы кучи.
        
2. **Когда используется несколько индексов или комбинации условий**
    
    - Запрос с `OR`, комбинация `AND`/`OR` по нескольким индексируемым полям, например:
        
        `WHERE col1 < 100 OR col2 > 200`
        
    - Между `Bitmap Index Scan`‑ами вставляются `BitmapAnd`/`BitmapOr`, а затем идёт `Bitmap Heap Scan` по объединённой битовой карте.
        
3. **Когда нужно минимизировать random I/O**
    
    - `Bitmap Heap Scan` читает **страницы кучи в порядке их физического расположения** (почти последовательно), а не в порядке ключей индекса, благодаря битовой карте.
        
    - Это снижает хаотичный I/O и увеличивает шанс попасть в кэш ОС.
        
>`Recheck Cond` — PostgreSQL повторно проверяет условие уже на страницах кучи (на случай, что индекс не учитывал все тонкости, например `VACUUM`/MVCC‑информацию).

Q: В каких случаях будет применён Index Only Scan для организации доступа к данным?
A: `Index Only Scan` применяется, когда **все данные запроса «лежат» в одном индексе**, и **Visibility Map доказывает**, что не нужно ходить в таблицу за дополнительной MVCC‑информацией.
	
**Основные условия**
	
1. **Запрос читает только поля, которые есть в индексе**
    
    - Все столбцы в `SELECT` и в `WHERE` (и `JOIN ON`, если есть) должны входить в один и тот же индекс: либо в ключ (`key`), либо в `INCLUDE`.
        
    - Например: индекс `(a, b, c)` + запрос `SELECT a, b FROM table WHERE c = ...` может использовать `Index Only Scan`.
        
2. **PostgreSQL «уверен», что строки видимы текущей транзакции**
    
    - Для MVCC‑видимости нужно проверить, что на странице не осталось «мертвых» или невидимых строк.
        
    - Для этого используется **Visibility Map**: если на странице флаг «все строки видимы», PostgreSQL может обойтись без чтения самой таблицы и использовать `Index Only Scan`.
        
3. **После `VACUUM` / `ANALYZE` видимость страниц обновлена**
    
    - Если таблица часто обновляется, а `VACUUM` почти не запускался, `Visibility Map` неполный и `Index Only Scan` будет реже выбираться, т.к. СУБД часто вынуждена читать heap‑строки для проверки видимости.

## Типичные случаи на практике

- Запросы только по ключевым/фильтрационным полям, где есть индекс, и не нужен `SELECT *` (или нужные столбцы затянуты в `INCLUDE`).
    
- Операции вроде `SELECT count(*) FROM table WHERE indexed_col = ...` по индексу, если он покрывает все нужные данные.
