# JOIN strategies - PostgreSQL


JOIN стратегию указать явно нельзя. 



![[Pasted image 20260214180431.png]]


- Для каждой строки **левой таблицы** последовательно просматриваются все строки **правой таблицы** на предмет совпадения по условию `ON`.
    
- Эффективен только для **маленьких таблиц** или когда у правой таблицы очень мало строк на каждую строку из левой.


---

![[Pasted image 20260214180453.png]]

- То же, что обычный `NESTED LOOP`, но для правой таблицы используется **индекс по полю из условия `ON`**.
    
- Вместо полного сканирования правой таблицы оптимизатор делает по **каждому ключу из левой таблицы поиск по индексу**, поэтому гораздо быстрее, если индекс хороший.


---

![[Pasted image 20260214180607.png]]

- Обе таблицы **предварительно сортируются** по полям условия `ON`.
    
- Затем строки обеих таблиц проходят **по одному проходу** (как сортировка слиянием): как только ключи совпали, строки «склеиваются».
    
- Хорошо для больших таблиц, если есть **индексы по полям JOIN** или данные уже отсортированы (например, из `ORDER BY` / `GROUP BY`).



---

**Когда нет индексов вообще (даже по id)**

![[Pasted image 20260215153515.png]]

- Для **меньшей таблицы** строится **хеш‑таблица** по полям условия `ON`.
    
- Затем строки **большей таблицы** проходят один раз, и для каждой строки вычисляется хеш и ищется соответствующий ключ в хеш‑таблице.
    
- Очень эффективен для **больших таблиц без нужных индексов**, особенно при `INNER JOIN`, но требует **дополнительной памяти**.
