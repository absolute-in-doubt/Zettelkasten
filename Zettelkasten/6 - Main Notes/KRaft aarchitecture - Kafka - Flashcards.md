Q: Для чего нужны KRaft контроллеры в Kafka? Кто может быть контроллером? Кикие данные они хранят и где?
A: Контроллеры хранят метаданные о состоянии кластера и назначают leader partitions. Контроллеры KRaft определяются в конфигурации (мы явно определяем какие поды будут контроллерами). Также контроллерами могут быть и обычные брокеры (совмещать работу с сообщениями и работу с метаданными), но в проде это не рекомендуется.
	
 Контроллеры держат метаданные кластера в журнале `__cluster_metadata` на диске и в памяти, а сам кворум контроллеров обеспечивает консистентное распространение изменений метаданных по нескольким нодам.
	
Какие данные хранит контроллер KRaft:
	
- Метаданные кластера: 
	- список топиков, их расположение (где какие партиции, где какие лидеры, в какой конкретно хост ходить)
	- конфигурации топиков и брокеров, 
	- партиции и распределение лидеров
    
- Журнал изменений: все операции над метаданными фиксируются как события в журнале кластера, которые реплицируются между контроллерами и считаются коммитнутыми после достижения кворума. 
    
- Снимки состояния (state machine): после применения изменений лидер контроллер периодически обновляет локальный снимок состояния и рассылает обновления остальным компонентам кластера.

Q: Опиши процесс отправки сообщения в Kafka. (Что делает Producer?)
A:    
1. Запрашивает метаданные у KRaft Controller: список топиков, их расположение (где какие партиции, где какие лидеры, в какой конкретно хост ходить)
2. Сериализует сообщение (byte array, json, protobuf)
3. Определяет, в какую партицию будет отправлять сообщения.
   Алгоритмы для выбора партиции:
	-  Explicit partition
	- Roud-robin
	- Key-defined
	- Sticky partitioner
4. Сжатие данных
   Алгоритмы сжатия (compression.codec):
	   - none
	   - GZip
	   - Snappy
	   - LZip4
	   - ZipSTD
5. Создания батча (группировка нескольких сообщений) для уменьшения накладных расходов отправки
	- `batch.size` = 16kb (by default) - размер батча (пихает в батч сообщения, пока не наберётся >= `batch.size`)
	- `linger.ms` = 5ms (by default) - интервал, через который автоматически будет закрыт batch. (Даже если не набрался нужный размер)
	
> **NB!** Если сообщение очень больше (> `batch.size`, оно может быть отправлено сразу - без создания batch.

Q: Опиши процесс получения сообщения из Kafka. (Что делает Consumer?)
A:    
1. Получение метаданных из контроеллера KRaft:
	 - состояние кластера
	 - состояние топиков
	 - состояние партиций
	
> **NB!** Какой partition будет читать Consumer - определяет сама Kafka (KRaft controller) при первом подключении Consumer-а в составе Consumer Group. Если кол-во Consumers > amt of partitions => те Consumers, что не успели получить свой partition, будут простаивать.
	
2. **Читает из partition сразу целый batch**
	- Kafka собирает батч из имеющихся сообщений
	- Consumer читает, сохраняет в буфер локально
	- обрабатывает из буфера по порядку
