# OIDC для межсервисного взаимодействия - System Design



Для сервис-сервис взаимодействия в Keycloak **не используются** потоки для пользователей (например, **Authorization Code Flow**). Вместо этого применяются:

---

### **Client Credentials Flow (основной для сервисов)**

- **Сценарий:** Сервис A вызывает Сервис B от своего имени (без участия пользователя).
    
- **Как работает:**
    
    1. Сервис A аутентифицируется у **Identity Provider** (например, Keycloak) с помощью **`client_id` + `client_secret`**.
    2. Получает **Access Token** (JWT).
    3. Отправляет запрос к Сервису B с заголовком:
        
        ```
        Authorization: Bearer <access_token>
        ```
        
    4. Сервис B **валидирует токен** (локально или через Introspection Endpoint).
- **Плюсы:**
    
    - Простота (нет пользовательского контекста).
    - Поддержка **scopes** (например, `service:read`, `service:write`).
- **Минусы:**
    
    - Токены **не содержат информации о пользователе** (только о сервисе).
