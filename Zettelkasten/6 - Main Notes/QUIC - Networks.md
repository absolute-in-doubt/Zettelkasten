# QUIC - Networks


> QUIC — это современный транспортный протокол поверх UDP, используемый HTTP/3. Он обеспечивает надёжную доставку, встроенное шифрование на основе TLS 1.3, мультиплексирование потоков и более быстрое установление соединения по сравнению с TCP. Благодаря независимым потокам QUIC устраняет проблему Head-of-Line Blocking, характерную для HTTP/2 поверх TCP.


---
## Зачем появился QUIC?

Проблема классического стека:

```text
HTTP/2
   ↓
TLS
   ↓
TCP
```

состоит в том, что:

1. Для установки соединения требуется несколько RTT:
    
    - TCP Handshake;
        
    - затем TLS Handshake.
	
>**RTT (Round-Trip Time)** — это время, которое требуется пакету, чтобы:
>
>1. дойти от отправителя до получателя;
>2. получить ответ;
>3. вернуться обратно.
>
>Проще говоря:
>
> **RTT — это время полного пути "туда и обратно".**


2. В TCP есть проблема **Head-of-Line Blocking**:  
    если потерялся один TCP-сегмент, все последующие данные задерживаются.
	

3. Изменять TCP сложно:  
    TCP реализован в ядре ОС, поэтому внедрение новых алгоритмов занимает годы.
    

QUIC был создан, чтобы решить эти проблемы.


---
### Установление нового соединения (1-RTT Handshake)

QUIC всегда использует встроенный TLS 1.3.

#### Шаг 1. Клиент отправляет Initial

```text
Клиент                                           Сервер
   |                                                  |
   |--- Initial ------------------------------------->|
   |     ClientHello (TLS)                            |
   |     Connection ID                                |
   |     Transport Parameters                         |
   |                                                  |
```

Поскольку клиент ещё не знает ключей шифрования сервера, пакет шифруется специальными Initial keys.

После получения этого пакета сервер:

- создаёт QUIC connection;
    
- извлекает ClientHello;
    
- начинает TLS handshake.
    
#### Шаг 2. Сервер отвечает

```text
Клиент                                           Сервер
   |                                                  |
   |<-- Initial --------------------------------------|
   |    ServerHello                                   |
   |                                                  |
   |<-- Handshake ------------------------------------|
   |    EncryptedExtensions                           |
   |    Certificate                                  |
   |    CertificateVerify                            |
   |    Finished                                     |
   |                                                  |
```

Одновременно сервер отправляет ACK на полученный Initial packet клиента.

Упрощённо:

```text
Server -> ACK(Client Initial)
```

На практике ACK обычно инкапсулируется в один из отправляемых пакетов.


#### Шаг 3. Клиент подтверждает handshake

После получения Finished клиент:

```text
Клиент                                           Сервер
   |--- Handshake ----------------------------------->|
   |    Finished                                      |
   |                                                  |
   |--- ACK ----------------------------------------->|
   |                                                  |
```

Клиент:

- завершает TLS handshake;
    
- начинает использовать application keys.
    

После получения Finished сервер считает handshake завершённым.

> [!warning]
> Обрати внимание: клиентский `Finished` обычно отправляется **вместе с первым HTTP-запросом**.

#### Полная диаграмма установки соединения

```text
Клиент                                               Сервер
   |                                                     |
   | Initial                                             |
   | ClientHello --------------------------------------->|
   |                                                     |
   |<----------------------------------- Initial         |
   |                                     ServerHello     |
   |                                                     |
   |<---------------------------------- Handshake        |
   |                         Certificate + Finished      |
   |                                                     |
   | Handshake                                           |
   | Finished ------------------------------------------>|
   |                                                     |
   | ACK ----------------------------------------------->|
   |                                                     |
   |================== Connection established ===========|
```

~={orange}В отличие от TCP:=~

```text
TCP: SYN -> SYN/ACK -> ACK
```

~={orange}в QUIC handshake и TLS происходят одновременно.=~


### Повторное подключение (0-RTT)

При следующем подключении клиент сразу отправляет:

```
Клиент                                  Сервер

ClientHello
+ Session Ticket
+ Early Data (HTTP Request) ----------->

<-------------------- ServerHello + Finished
```

Например:

```
GET /index.html
```

отправляется **в первом же пакете**, не ожидая ответа сервера.

Именно поэтому это называется **0-RTT**:

> клиенту не нужно ждать ни одного RTT, чтобы начать передачу application data.


---
### Передача данных

После handshake обе стороны ~={orange}могут одновременно передавать данные=~.

QUIC является ~={orange}full-duplex=~.

```text
Клиент                                           Сервер

STREAM #1 data ------------------------------->

<-------------------------------- STREAM #2 data

ACK ------------------------------------------>

<------------------------------------------ ACK
```

Очень важная особенность:

ACK в QUIC:

- не отправляется немедленно на каждый пакет;
    
- может подтверждать сразу множество пакетов;
    
- ~={orange}может быть объединён с обычными данными=~.
    

Например:

```text
Client -> packets 100,101,102

Server -> ACK {100-102}
```


---
### Обнаружение потери пакетов

Если ACK не приходит вовремя:

```text
Packet #200 ----------------------------X

ACK отсутствует
```

отправитель:

- считает пакет потерянным;
    
- повторно отправляет данные.
    

**Важно:**

QUIC повторно передаёт **данные**, а не потерянный пакет целиком.

В TCP повторная передача привязана к **последовательности байтов**. В QUIC же есть разделение между:

1. **Packet Number** — номером пакета, используемым для обнаружения потерь.
2. **Данными приложения** (frames в потоках).

Из-за этого QUIC никогда не отправляет повторно **тот же самый пакет** с тем же номером.

Например, клиент отправляет:

```
Packet #10:
  STREAM frame:
      Stream 1, Offset 0, Length 1000
```

Пакет #10 потерялся.

В TCP можно условно думать так:

> «Переотправим тот же сегмент ещё раз».

В QUIC происходит иначе:

```
Packet #25:
  STREAM frame:
      Stream 1, Offset 0, Length 1000
```

То есть:

- данные те же (`Stream 1`, байты `0-999`);
- но номер пакета уже новый (`#25`, а не `#10`).

Старый Packet #10 считается навсегда потерянным и больше никогда не используется.


---
### Migration (смена IP)

Одно из уникальных свойств QUIC.

Например:

```text
WiFi -> LTE
```

TCP:

```text
Connection broken
```

QUIC:

```text
Connection ID сохраняется
```

```text
Client (WiFi IP)
        |
        v
Client (LTE IP)

Connection ID = тот же
```

Сервер продолжает использовать существующее соединение.

---

### Graceful shutdown

Корректное закрытие соединения происходит через специальные QUIC frames.

Обычно используется:

```text
CONNECTION_CLOSE
```

Диаграмма:

```text
Клиент                                           Сервер

CONNECTION_CLOSE ------------------------------->

<------------------------------- ACK

Connection terminated
```

В отличие от TCP:

```text
FIN
ACK
FIN
ACK
```

четырёхстороннего handshake здесь нет.

---

# 6. Abrupt termination

При серьёзной ошибке:

```text
TLS error
Protocol violation
Flow control violation
```

отправляется:

```text
CONNECTION_CLOSE(error_code)
```

после чего соединение немедленно завершается.


# Что нужно знать для интервью

Обычно достаточно помнить:

1. QUIC работает поверх UDP.
    
2. QUIC совмещает transport + TLS handshake.
    
3. QUIC устанавливает соединение за 1 RTT.
    
4. QUIC использует ACK и retransmission.
    
5. QUIC повторно отправляет данные, а не пакеты.
    
6. QUIC поддерживает migration между IP.
    
7. QUIC использует CONNECTION_CLOSE вместо FIN/ACK.

## Какие функции предоставляет QUIC?

### 1. Надёжная доставка

Как и TCP, QUIC:

- нумерует данные;
    
- использует ACK;
    
- переотправляет потерянные пакеты.
    

---

### 2. Встроенное шифрование

В QUIC шифрование обязательно.

QUIC фактически включает возможности TLS 1.3 в сам протокол.

Поэтому:

```text
HTTP/3
   ↓
QUIC (включает TLS 1.3)
   ↓
UDP
```

Отдельного TLS-слоя больше нет.

---

### 3. Мультиплексирование потоков

Одно QUIC-соединение содержит множество независимых потоков:

```text
QUIC connection
├── Stream #1
├── Stream #2
├── Stream #3
```

Если в одном потоке произошла потеря пакета, остальные потоки продолжают работать.

---

## Главное преимущество перед TCP

### Нет Transport Head-of-Line Blocking

В HTTP/2:

```text
HTTP/2 streams
       ↓
один TCP stream
```

Потеря одного TCP-сегмента блокирует все HTTP-потоки.

В QUIC:

```text
Stream #1 потерял пакет
```

не влияет на:

```text
Stream #2
Stream #3
```

---

## 0-RTT и быстрое установление соединения

TCP + TLS:

```text
TCP Handshake
TLS Handshake
```

требует несколько RTT.

QUIC способен:

- установить новое соединение за 1 RTT;
    
- повторное соединение — за 0 RTT.
    

Это особенно важно для мобильных устройств.
